# ClickHouse I/O 流程深度解析

> 基于 ClickHouse 源码 (ClickHouse-master) 分析，聚焦 I/O 层的 Buffer 链、压缩、缓存、异步与预取机制

## 二、Disk 磁盘抽象

```

│                                                               │

│  │                                                           │

│  │                                                           │

│  │   → unique_ptr<WriteBufferFromFileBase>          │      │
│  │                                                 │      │
│  │ createTransaction()                             │      │
│  │   → DiskTransactionPtr                         │      │

│                                                               │

│  DiskSelector: 从 XML 配置解析 <disks> 段                    │
│  DiskFactory: 注册 + 创建磁盘实例 (单例工厂)                  │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

## 五、Buffer 链详解

```
┌───────────────────────────────────────────────────────────────┐
│                  ReadBuffer 链式组合                           │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  读取一个压缩列文件的完整 Buffer 链:                           │
│                                                               │
│  用户代码 (MergeTreeReader)                                   │
│       │                                                       │
│       ▼                                                       │
│  ┌──────────────────────────────────────┐                    │
│  │ CachedCompressedReadBuffer            │                    │

## 六、缓存体系

```
┌───────────────────────────────────────────────────────────────┐
│                  三级缓存体系                                   │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  Level 1: MarkCache (.mrk 文件缓存)                            │
│  ┌────────────────────────────────────────────────────┐      │
│  │                                                    │      │
│  │  缓存内容: .mrk / .mrk2 文件中所有 Mark 的集合      │      │
│  │  Cache Key: SipHash(disk_name + ":" + mrk_path)    │      │
│  │  Cache Value: MarksInCompressedFile                │      │
│  │               (内存压缩格式, ~3 字节/Mark)          │      │
│  │  淘汰策略: SLRU (Segmented LRU)                    │      │
│  │  大小控制: mark_cache_size (默认 5GB)               │      │
│  │                                                    │      │
│  │  命中时: 直接返回 Mark 数组, 无磁盘 I/O             │      │
│  │  未命中: 读磁盘 → 解析 → 压缩存储 → 放入缓存        │      │
│  │                                                    │      │
│  │  异步加载:                                        │      │
│  │    MergeTreeMarksLoader 支持异步加载               │      │
│  │    load_marks_asynchronously = true 时             │      │
│  │    → 提交到 LoadMarksThreadpool                    │      │
│  │    → 不阻塞查询线程                                │      │
│  │                                                    │      │
│  │  内存优化:                                        │      │
│  │    MarksInCompressedFile 使用 bit-pack 压缩        │      │
│  │    每 256 个 Mark 一块, 差值编码                    │      │
│  │    例: 100 万 Mark → ~3MB (vs 原始 16MB)           │      │
│  │                                                    │      │
│  └────────────────────────────────────────────────────┘      │
│                                                               │
│  Level 2: UncompressedCache (解压块缓存)                       │
│  ┌────────────────────────────────────────────────────┐      │
│  │                                                    │      │
│  │  缓存内容: 已解压的列数据块                         │      │
│  │  Cache Key: SipHash(path + offset_in_file)         │      │
│  │             (每个压缩块一个缓存条目)                │      │
│  │  Cache Value: UncompressedCacheCell                │      │
│  │    { Memory<JemallocCacheAllocator> data;           │      │

│  │  淘汰策略: SLRU                                     │      │
│  │  大小控制: uncompressed_cache_size (默认 8GB)       │      │
│  │                                                    │      │
│  │  命中时: working_buffer 直接指向缓存内存 (零拷贝)    │      │
│  │          跳过 read + decompress → 极快              │      │
│  │  未命中: read 压缩块 → CityHash 校验 → decompress   │      │
│  │          → 存入缓存 → working_buffer 指向缓存       │      │
│  │                                                    │      │
│  │  内存分配器:                                      │      │
│  │    使用 JemallocCacheAllocator                     │      │
│  │    → 利用 jemalloc introspection API 精确统计      │      │
│  │      缓存实际占用内存                               │      │
│  │                                                    │      │
│  │  热点场景:                                        │      │
│  │    频繁查询的列 → 解压块常驻缓存 → 避免 CPU 解压     │      │
│  │    冷数据 → 自然淘汰 → 不浪费内存                   │      │
│  │                                                    │      │
│  └────────────────────────────────────────────────────┘      │
│                                                               │
│  Level 3: 页缓存 (Page Cache)                                 │
│  ┌────────────────────────────────────────────────────┐      │
│  │                                                    │      │
│  │  操作系统页缓存:                                   │      │
│  │    read()/pread() 自动使用 Linux Page Cache        │      │
│  │    热数据页驻留内存, 冷数据页自动回收               │      │
│  │    posix_fadvise(DONTNEED) 主动释放                │      │
│  │                                                    │      │
│  │  O_DIRECT 模式 (可选):                             │      │
│  │    direct_io_threshold = N                        │      │
│  │    当预估读取 > N 字节时, 打开 O_DIRECT            │      │
│  │    跳过页缓存, 直接磁盘 I/O                        │      │
│  │    适合大扫描, 避免缓存污染                        │      │
│  │                                                    │      │
│  │  用户态页缓存 (可选):                              │      │
│  │    CachedInMemoryReadBufferFromFile                │      │
│  │    page_cache → 管理自己的内存页缓存                │      │
│  │    适合远程存储 (S3), 避免重复网络请求              │      │
│  │    readBigAt() 可零拷贝返回缓存页                   │      │
│  │                                                    │      │
│  └────────────────────────────────────────────────────┘      │
│                                                               │
│  缓存命中对性能的影响:                                        │
│                                                               │

│  ─────────────────────────────────────────────────────       │
│  首次查询          Miss       Miss              Miss         │
│  同查询重跑        Hit        Hit (可能)         Hit (可能)   │

## 八、Seek 机制详解

```
┌───────────────────────────────────────────────────────────────┐
│                  Seek 路径详解                                  │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  从 Mark 2 跳到 Mark 5:                                       │
│                                                               │
│  Step 1: 获取 Mark                                            │
│  ┌──────────────────────────────────────────────────┐        │
│  │ marks_getter->getMark(5, column_idx)             │        │
│  │                                                  │        │
│  │ 返回: MarkInCompressedFile {                     │        │
│  │   offset_in_compressed_file = 45678,             │        │
│  │   offset_in_decompressed_block = 123             │        │
│  │ }                                               │        │
│  │                                                  │        │
│  │ 含义:                                           │        │
│  │   压缩文件的第 45678 字节处有一个压缩块            │        │
│  │   解压后, 从第 123 字节开始是 Granule 5 的数据    │        │
│  └──────────────────────────────────────────────────┘        │
│                                                               │
│  Step 2: Lazy Seek (关键优化)                                 │
│  ┌──────────────────────────────────────────────────┐        │
│  │                                                  │        │
│  │ CompressedReadBufferFromFile::seek()              │        │
│  │                                                  │        │
│  │ Fast Path 1: 目标在当前 working_buffer 内?        │        │
│  │   compressed_pos == 45678 &&                     │        │
│  │   decompressed_offset 在 buffer 范围内             │        │
│  │   → 只移动 pos 指针, 零开销!                     │        │
│  │                                                  │        │
│  │ Fast Path 2: pread 模式?                         │        │
│  │   → 无需 lseek, 直接更新 file_offset              │        │
│  │   → 下次 pread() 用新 offset                     │        │
│  │                                                  │        │
│  │ Slow Path: 需要 lseek                            │        │
│  │   → lseek(fd, 45678, SEEK_SET)                  │        │
│  │   → 丢弃 working_buffer                          │        │
│  │   → 存储 decompressed_offset = 123               │        │
│  │   → 实际数据在下次 nextImpl() 时才加载            │        │
│  │                                                  │        │
│  └──────────────────────────────────────────────────┘        │
│                                                               │
│  Step 3: 延迟加载 (Lazy Load)                                 │
│  ┌──────────────────────────────────────────────────┐        │
│  │                                                  │        │
│  │ 下次 nextImpl() 调用时:                           │        │
│  │                                                  │        │
│  │ 1. readCompressedData():                          │        │
│  │    从 offset 45678 读取压缩块头                    │        │
│  │    CityHash128 校验                               │        │
│  │    读取 compressed_size 和 decompressed_size       │        │
│  │                                                  │        │
│  │ 2. decompress():                                 │        │
│  │    解压到 memory buffer                           │        │
│  │    memory.resize(decompressed_size + SIMD_padding)│        │
│  │                                                  │        │
│  │ 3. 跳过已读取的字节:                              │        │
│  │    pos = working_buffer + 123  (decompressed_offset)│        │
│  │    → 直接从 Granule 5 的数据开始读取               │        │
│  │                                                  │        │
│  │ readBig() 优化 (大批量读取):                      │        │
│  │    如果解压块能完全放入目标 buffer:                │        │
│  │    → 直接解压到目标 buffer, 避免中间拷贝           │        │
│  │                                                  │        │
│  └──────────────────────────────────────────────────┘        │
│                                                               │
│  连续 Mark 读取 vs 随机 Mark 读取:                             │
│                                                               │
│  连续读取 (Mark 5 → 6 → 7):                                  │
│  ┌──────────────────────────────────────────────────┐        │
│  │ Mark 5: seek + read + decompress                  │        │
│  │ Mark 6: 可能在同一压缩块 → Fast Path (无 seek)    │        │
│  │ Mark 7: 可能需要新压缩块 → seek + read + decompress│        │
│  │                                                  │        │
│  │ 压缩块通常跨越多个 Granule (64KB~1MB per block)  │        │
│  │ → 连续读取大部分时间不需要 seek                   │        │
│  └──────────────────────────────────────────────────┘        │
│                                                               │
│  随机读取 (Mark 2 → 50 → 3):                                 │
│  ┌──────────────────────────────────────────────────┐        │
│  │ Mark 2:  seek(12345) + read + decompress          │        │
│  │ Mark 50: seek(98765) + read + decompress  ← 必须seek│        │
│  │ Mark 3:  seek(15000) + read + decompress  ← 必须seek│        │
│  │                                                  │        │
│  │ 每次 seek: lseek + 丢弃 buffer + 重新解压        │        │
│  │ → 随机读取性能差 (LSM/MergeTree 的固有代价)      │        │
│  │ → UncompressedCache 可缓解 (热点块不重复解压)     │        │
│  └──────────────────────────────────────────────────┘        │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

## 十三、设计精髓总结

```
1. 装饰器模式的 Buffer 链
   每层 Buffer 只做一件事:
   ReadBufferFromFile (系统调用)
   → CompressedReadBuffer (解压)
   → CachedCompressedReadBuffer (缓存)
   → MergeTreeReaderStream (Mark 定位)
   通过组合不同 Buffer 实现各种 I/O 模式

2. Lazy Seek (延迟定位)
   seek 不立即执行 I/O, 只记录目标位置
   下次 nextImpl() 时才真正加载
   → 如果连续 Mark 在同一压缩块, 完全跳过 seek

3. 零拷贝设计
   - UncompressedCache 命中: working_buffer 直接指向缓存内存
   - readBig(): 解压直接到目标 buffer, 避免中间拷贝
   - 压缩块在源 buffer 内时: 直接引用, 不拷贝

4. 三级缓存协作
   MarkCache (索引元数据) → 减少磁盘 I/O
   UncompressedCache (解压数据) → 减少 CPU 解压
   Page Cache (操作系统) → 减少磁盘延迟
   三者互补, 各司其职

5. I/O 与计算重叠
   posix_fadvise / io_uring 双缓冲
   预取在任务分配时就开始, 不等线程就绪
   多列并行 I/O, 内核/磁盘并行调度

6. 远程存储适配
   BoundedReadBuffer: 精确范围请求, 节省带宽
   ParallelReadBuffer: 多流并行下载, 突破单流限制
   磁盘缓存: S3 → 本地 SSD → 读取, 消除网络延迟

7. 自适应反压
   监控 I/O 吞吐和延迟
   慢读时自动降低并发, 恢复时逐步增加
   避免磁盘过载导致全面性能下降
```
