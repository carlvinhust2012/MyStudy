```

## 三、列式存储格式

```
┌───────────────────────────────────────────────────────────────┐
│                  列式存储详细格式                               │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  表: CREATE TABLE t (id UInt32, name String, value Float64)   │
│      ORDER BY id                                             │
│                                                               │
│  数据行:                                                      │
│    ┌────┬──────────┬───────┐                                  │
│    │ id │   name   │ value │                                  │
│    ├────┼──────────┼───────┤                                  │
│    │ 1  │ "alice"  │ 10.5  │                                  │
│    │ 3  │ "bob"    │ 20.3  │                                  │
│    │ 5  │ "carol"  │ 15.7  │  ← 一个 Granule (默认 8192 行)   │
│    │ 7  │ "dave"   │ 30.1  │                                  │
│    │ .. │   ...    │  ...  │                                  │
│    │8191│ "zzz"    │ 99.9  │                                  │
│    ├────┼──────────┼───────┤                                  │
│    │8192│ "aaa"    │ 11.0  │  ← 下一个 Granule               │
│    │ .. │   ...    │  ...  │                                  │
│    └────┴──────────┴───────┘                                  │
│                                                               │
│  磁盘上 (Wide 格式, 每列独立文件):                              │
│                                                               │
│  id.bin (LZ4 压缩):                                          │
│  ┌──────────────────────────────────────────┐                 │
│  │ Granule 0: [1,3,5,7,...,8191]  → 压缩块 │ ← 一次解压      │
│  │ Granule 1: [8192,8194,...]     → 压缩块 │                  │
│  │ Granule 2: [...]              → 压缩块 │                  │
│  └──────────────────────────────────────────┘                 │
│                                                               │
│  name.bin (LZ4 + ZSTD 双压缩):                               │
│  ┌──────────────────────────────────────────┐                 │
│  │ Granule 0: ["alice","bob",...,"zzz"]     │                  │
│  │ Granule 1: ["aaa",...]                   │                  │
│  └──────────────────────────────────────────┘                 │
│                                                               │
│  value.bin (LZ4 压缩):                                       │
│  ┌──────────────────────────────────────────┐                 │
│  │ Granule 0: [10.5,20.3,15.7,...,99.9]    │                  │
│  │ Granule 1: [11.0,...]                    │                  │
│  └──────────────────────────────────────────┘                 │
│                                                               │
│  Mark 文件 (id.mrk2, 自适应粒度):                             │
│  ┌─────────────────────────────────────────────────────┐      │
│  │ Mark 0: (压缩偏移, 解压偏移, 行数)                    │      │
│  │         (0,    0,    8192)                           │      │
│  │ Mark 1: (压缩偏移, 解压偏移, 行数)                    │      │
│  │         (12345, 678,  4096)  ← 自适应, 行数可变       │      │
│  │ Mark 2: ...                                         │      │
│  │                                                     │      │
│  │ 每个 Mark = 24 字节 (3 × UInt64)                    │      │
│  │ 作用: 快速定位到 Granule 在压缩文件中的位置           │      │
│  │  → 读取 Mark N → seek 到压缩偏移 → 解压 → 跳到行    │      │
│  └─────────────────────────────────────────────────────┘      │
│                                                               │
│  Compact 格式 (小 part 用, 单文件):                            │
│  ┌──────────────────────────────────────────────┐             │
│  │ data.bin:                                    │             │
│  │   Granule 0: [id列数据][name列数据][value列数据]│            │
│  │   Granule 1: [id列数据][name列数据][value列数据]│            │
│  │ data.mrk3: (所有列的偏移信息)                 │             │
│  └──────────────────────────────────────────────┘             │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 索引粒度 (Index Granularity)

```
┌───────────────────────────────────────────────────────────────┐
│                  索引粒度机制                                   │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  固定粒度 (默认):                                              │
│    index_granularity = 8192                                   │
│    每 8192 行生成一个 Mark, 主键索引记录一个条目               │
│                                                               │
│  自适应粒度 (index_granularity_bytes 启用):                    │
│    目标: 每个压缩块约 index_granularity_bytes (默认 10MB)     │
│    原因: 不同行大小差异大 (String 列 vs UInt64 列)            │
│    效果: 压缩块大小更均匀, 解压效率更高                       │
│                                                               │
│  对比:                                                        │
│                                                               │
│  固定粒度:                                                    │
│  ┌────────┬────────┬────────┬────────┐                       │
│  │ 8192行 │ 8192行 │ 8192行 │ 8192行 │                       │
│  │  1MB   │ 50MB   │  2MB   │ 100MB  │ ← 压缩块大小不均匀    │
│  └────────┴────────┴────────┴────────┘                       │
│                                                               │
│  自适应粒度:                                                  │
│  ┌───────┬───────┬───────┬───────┐                           │
│  │ 4096行│ 8192行│ 16384行│ 8192行│                           │
│  │ ~10MB │ ~10MB │ ~10MB │ ~10MB │ ← 压缩块大小均匀          │
│  └───────┴───────┴───────┴───────┘                           │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

## 四、稀疏主键索引

```
┌───────────────────────────────────────────────────────────────┐
│                  稀疏主键索引                                   │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  主键 ≠ 唯一键 (ClickHouse 不保证唯一性)                      │
│  主键 = 排序键的前缀 (用于加速范围查询)                        │
│                                                               │
│  primary.idx 结构:                                            │
│  ┌──────────────────────────────────────────┐                 │
│  │  记录: 每 index_granularity 行一个条目    │                 │
│  │  内容: 该行所有主键列的值                  │                 │
│  │  大小: 非常小 (8192 行只记 1 个点)        │                 │
│  └──────────────────────────────────────────┘                 │
│                                                               │
│  示例: ORDER BY (event_date, id)                              │
│                                                               │
│  primary.idx:                                                 │
│  ┌─────────────┬──────────┐                                   │
│  │ event_date  │    id    │                                   │
│  ├─────────────┼──────────┤                                   │
│  │ 2025-01-01  │    1     │  ← 第 0 行 (Mark 0)              │
│  │ 2025-01-01  │  8192    │  ← 第 8192 行 (Mark 1)          │
│  │ 2025-01-02  │    5     │  ← 第 16384 行 (Mark 2)         │
│  │ 2025-01-03  │  12000   │  ← 第 24576 行 (Mark 3)         │
│  │ ...         │   ...    │                                   │
│  └─────────────┴──────────┘                                   │
│                                                               │
│  查询: WHERE event_date = '2025-01-02'                        │
│                                                               │
│  索引查找过程 (KeyCondition):                                  │
│  ┌─────────────────────────────────────────────────────┐      │
│  │ 1. 二分查找 primary.idx                              │      │
│  │    找到 Mark 2 (2025-01-02) 和 Mark 3 (2025-01-03)  │      │
│  │                                                     │      │
│  │ 2. 确定 Mark 范围: [2, 3)                           │      │
│  │    → 只读 Mark 2 对应的 Granule                     │      │
│  │    → 跳过 75% 的数据                                 │      │
│  │                                                     │      │
│  │ 3. 读取 Mark 2 对应的数据:                           │      │
│  │    从 .mrk2 读取偏移 → seek 到 .bin → 解压 → 过滤   │      │
│  └─────────────────────────────────────────────────────┘      │
│                                                               │
│  查询: WHERE id = 5000                                        │
│  问题: id 不是主键的前缀 (event_date 才是)                    │
│  → 无法利用 primary.idx, 必须全表扫描                        │
│  → 解决: ORDER BY (id, ...) 把常用过滤列放前面               │
│                                                               │
│  KeyCondition 类 (src/Storages/MergeTree/KeyCondition.h):     │
│    把 WHERE 条件转为逆波兰表达式 (RPN)                       │
│    支持:                                                      │
│      - 范围比较: >, <, >=, <=, =, !=                         │
│      - IN / NOT IN                                           │
│      - IS NULL / IS NOT NULL                                  │
│      - 单调函数链: toDate(timestamp) > '...'                 │
│      - 多维超矩形: (x1,a1) < (col1,col2) < (x2,a2)         │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 多级索引过滤

```
┌───────────────────────────────────────────────────────────────┐
│                  多级索引过滤 (层层裁剪)                        │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  查询: SELECT * FROM t WHERE date = '2025-01-15'             │
│        AND user_id = 12345                                    │
│                                                               │
│  过滤层级:                                                    │
│                                                               │
│  Level 1: 分区裁剪 (Partition Pruning)                        │
│  ┌──────────────────────────────────────────────────┐        │
│  │ Part: 202501  → minmax: [2025-01-01, 2025-01-31] │        │
│  │ Part: 202502  → minmax: [2025-02-01, 2025-02-28] │        │
│  │ Part: 202503  → minmax: [2025-03-01, 2025-03-31] │        │
│  │                                                  │        │
│  │ WHERE date = '2025-01-15'                        │        │
│  │ → 只保留 202501 分区                              │        │
│  │ → 淘汰 2/3 的 Part                               │        │
│  └──────────────────────────────────────────────────┘        │
│                     │                                         │
│  Level 2: 主键索引 (Primary Index)                            │
│  ┌──────────────────────────────────────────────────┐        │
│  │ Part 202501 的 primary.idx:                       │        │
│  │   Mark 0: (2025-01-01, 1)                        │        │
│  │   Mark 1: (2025-01-05, 8000)                     │        │
│  │   Mark 2: (2025-01-10, 16000)   ← 可能包含        │        │
│  │   Mark 3: (2025-01-15, 24000)   ← 命中!           │        │
│  │   Mark 4: (2025-01-20, 32000)                    │        │
│  │   Mark 5: (2025-01-25, 40000)                    │        │
│  │                                                  │        │
│  │ → 保留 Mark [2, 5) 范围                           │        │
│  │ → 淘汰 50% 的 Granule                            │        │
│  └──────────────────────────────────────────────────┘        │
│                     │                                         │
│  Level 3: 跳数索引 (Skip Index, 可选)                         │
│  ┌──────────────────────────────────────────────────┐        │
│  │ 假设 user_id 列有 BloomFilter 索引:               │        │
│  │   Granule [2,3): bloom_check(12345) = false       │        │
│  │   Granule [3,4): bloom_check(12345) = true  ← 可能│        │
│  │   Granule [4,5): bloom_check(12345) = false       │        │
│  │                                                  │        │
│  │ → 只保留 Mark [3, 4)                              │        │
│  │ → 再淘汰 2/3 的 Granule                          │        │
│  └──────────────────────────────────────────────────┘        │
│                     │                                         │
│  Level 4: 数据读取 + 过滤                                     │
│  ┌──────────────────────────────────────────────────┐        │
│  │ 只读 1 个 Granule 的数据                          │        │
│  │ → 实际扫描 8192 行                               │        │
│  │ → WHERE 过滤出 user_id = 12345 的行               │        │
│  │ → 向量化执行 (SIMD)                              │        │
│  └──────────────────────────────────────────────────┘        │
│                                                               │
│  总过滤效果:                                                   │
│    3 个分区 → 1 个分区 (淘汰 67%)                             │
│    100 个 Granule → 3 个 Granule (淘汰 97%)                  │
│    3 个 Granule → 1 个 Granule (淘汰 67%)                    │
│    最终只读原始数据的 ~0.3%                                   │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

## 五、数据写入流程

```
┌───────────────────────────────────────────────────────────────┐
│                  INSERT 写入时序流程                             │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  SQL: INSERT INTO t VALUES (1, 'alice', 10.5), ...           │
│                                                               │
│  Client          MergeTreeSink        MergeTreeDataWriter     │
│    │                  │                       │               │
│    │── INSERT SQL ──▶│                       │               │

├───────────────────────────────────────────────────────────────┤
│                                                               │

│  └────────────────────────────────────────────────────┘      │
│                     │                                         │
│  合并执行 (MergeTask):                                        │
│  ┌────────────────────────────────────────────────────┐      │
│  │                                                    │      │
│  │  Source Parts:                Target Part:          │      │

│  - 合并层级 (level) 逐渐增加                                 │

## 十、设计精髓总结

```
1. LSM-Tree 变体
   ClickHouse 的 MergeTree 不是传统 B+ Tree, 也不是纯 LSM-Tree
   而是两者的混合:
   - 像 LSM-Tree: 写入 Append-Only, 后台合并
   - 像 B+ Tree: 数据全局有序, 支持范围查询
   - 区别于 LSM: 没有 MemTable, 直接写磁盘有序文件

2. 稀疏索引 vs 密集索引
   传统数据库: 每行一个索引条目 (B+ Tree 叶节点)
   ClickHouse: 每 8192 行一个索引条目
   → 索引极小, 可全部装入内存
   → 代价: 找到 Granule 后还需要扫描 Granule 内部

3. 列式存储的优势
   SELECT a, b FROM t WHERE c = x
   - 只读 c 列 (过滤) + a, b 列 (返回)
   - 不读其他列 → IO 放大极小
   - 同列连续存储 → 压缩比极高
   - 同列连续存储 → SIMD 向量化执行

4. 零 WAL 设计
   传统数据库: 写 WAL → 写内存 → 异步刷盘
   ClickHouse: 直接写有序文件 (不需要 WAL)
   → 写入延迟 = 排序 + 写磁盘
   → 代价: 插入单行效率低 (适合批量写入)

5. 多级过滤, 层层裁剪
   分区 → Part minmax → 主键索引 → 跳数索引 → 数据过滤
   每一层淘汰大量无关数据, 最终只读必要数据

6. 合并的写放大权衡
   base 参数控制合并激进程度
   base 大 → 写放大小, Part 多, 查询慢
   base 小 → 写放大大, Part 少, 查询快
   默认 base=5 是一个平衡点
```
