# SCM（Storage Class Memory，存储级内存）技术分析

## 1. 一句话概括

SCM 是一类介于传统 DRAM 和 NVMe SSD 之间的持久化存储介质，兼具接近内存的访问速度和断电不丢的特性。

## 2. 三层存储体系中的定位

```
┌─────────────────────────────────────────────────────────────────┐
│                       存储层次金字塔                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│          DRAM              SCM              NVMe SSD            │
│         (内存)          (存储级内存)         (固态硬盘)            │
│                                                                 │
│  延迟:  ~100ns           ~300ns           ~10-100us             │
│  容量:  GB 级            TB 级             TB 级                 │
│  持久化: 否               是                是                   │
│  寻址:  字节级            字节级            块级                  │
│  价格:  最贵             较贵              便宜                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

SCM 填补了 DRAM（快但不持久、容量小）和 SSD（持久但慢）之间的空白。

## 3. 访问延迟对比

| 指标 | DRAM | SCM | NVMe SSD | SATA SSD | HDD |
|---|---|---|---|---|---|
| 读取延迟 | ~100ns | ~300ns | ~10-100us | ~100us | ~10ms |
| 写入延迟 | ~100ns | ~1us | ~10-100us | ~100us | ~10ms |
| 带宽 | ~50GB/s | ~30GB/s | ~3-7GB/s | ~500MB/s | ~200MB/s |
| 单条最大容量 | 64-128GB | 512GB-1TB | 8-30TB | 4TB | 20TB |
| 持久化 | 否 | 是 | 是 | 是 | 是 |
| 字节寻址 | 是 | 是 | 否（块设备） | 否（块设备） | 否（块设备） |

## 5. 核心优点

### 5.1 字节级寻址

可以像 DRAM 一样通过 CPU 指令直接 load/store，不需要走块设备的 read/write 系统调用，无内核开销。

```
传统 SSD 访问路径（块设备）:
  应用程序 → read()/write() 系统调用 → 内核 VFS → 块设备驱动 → NVMe 控制器 → SSD
  延迟: ~10-100us

SCM 访问路径（字节寻址）:
  应用程序 → CPU load/store 指令 → 内存控制器 → SCM
  延迟: ~300ns（快 30-300 倍）
```

### 5.2 持久化

断电数据不丢失，这是 SCM 相对 DRAM 最本质的区别。

### 5.3 延迟接近 DRAM

约 300ns，是 NVMe SSD 的 1/100，是 HDD 的 1/100000。

### 5.4 大容量

单条可达 512GB-1TB（vs DDR4 单条 64GB），适合存大量元数据和热数据。

### 5.5 支持 DAX（Direct Access）

应用程序可以直接 mmap 映射 SCM，绕过 Page Cache，实现零拷贝访问。

```
DAX 模式:
  应用程序地址空间 → mmap → SCM 物理地址（直接映射，无 Page Cache 中间层）

非 DAX 模式（普通 SSD）:
  应用程序地址空间 → Page Cache → 块设备层 → SSD
```

## 6. 核心缺点

### 6.1 写入性能不对称

读延迟低（~300ns），但写延迟明显更高（~1us），且写带宽低于读带宽。原因是 SCM 需要将数据从 CPU Cache 刷到 SCM 介质。

### 6.2 需要手动持久化指令

写入数据必须手动执行 `clwb`（Cache Line Write Back）+ `sfence`，否则数据可能只停留在 CPU Cache 中，断电会丢。

```
CPU Cache 层次与持久化:

  CPU L1 Cache
  CPU L2 Cache
  CPU L3 Cache        ← 数据可能停留在这里，断电丢失
  ──────────────────── clwb + sfence 刷过这条线才算持久化
  SCM 物理介质          ← 数据安全，断电不丢
```

关键指令:
- `clwb addr` — 将指定 cache line 写回到 SCM
- `sfence` — 确保之前的 clwb 都已完成（内存屏障）
- `pccommit`（旧版 Optane）— 提交持久化域

### 6.3 容量有限且价格高

单位价格是 NVMe SSD 的 5-10 倍，不适合存储大量冷数据。

### 6.4 写入耐久性有限

写入耐久性（Endurance）低于 DRAM，远低于 SSD。Intel Optane DC PMem 的耐久性约为:
- App Direct 模式: 约 3 PBW（全盘写入总容量）
- 远低于企业级 SSD 的 10-30 DWPD

### 6.5 生态不成熟

- Intel Optane 已停产（2022 年宣布）
- CXL 方案还在发展中
- 软件生态（PMDK 等）还在完善
- 不同厂商产品 API 可能不兼容

### 6.6 对软件要求高

需要开发者显式管理持久化语义，不像 SSD 那样透明使用。需要使用 PMDK 等库来处理:
- 事务（tx_alloc, tx_free）
- 日志（undo log, redo log）
- 刷盘（clwb + sfence）
- 恢复（pool walk, log replay）

## 7. DAOS 中的 SCM 使用方式

DAOS 采用 SCM + NVMe 双介质架构，各取所长:

```
┌───────────────────────────────────────────────────────────┐
│                        DAOS 双介质架构                      │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  SCM（存储级内存）                                          │
│  - VOS 元数据（对象索引、KV 存储）                            │
│  - I/O 管理结构（VEA 分配器）                                │
│  - 热数据缓存                                               │
│  - WAL（预写日志）                                          │
│                                                           │
│  NVMe SSD                                                 │
│  - 大对象数据存储                                           │
│  - 冷数据                                                  │
│  - MD-on-SSD（元数据备份）                                   │
│                                                           │
│  选择标准: 需要低延迟随机访问的放 SCM，大批量顺序数据放 SSD     │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

## 8. SCM 适用场景总结

| 场景 | 适合 SCM | 原因 |
|---|---|---|
| 元数据存储 | 适合 | 需要低延迟随机访问，数据量不大 |
| KV 存储索引 | 适合 | 字节寻址，B-tree 节点直接访问 |
| 日志系统（WAL） | 适合 | 追加写，延迟敏感 |
| 大文件存储 | 不适合 | 成本高，容量有限，放 SSD 更合适 |
| 顺序批量写入 | 不适合 | 无法充分利用 SCM 低延迟优势 |
| 计算中间结果 | 不适合 | DRAM 更快，如果不需要持久化 |

## 9. 总结

**SCM = 接近内存速度 + 断电不丢 + 字节寻址**，代价是需要开发者手动管理持久化、价格高、写入性能不对称。在 DAOS 等分布式存储系统中，SCM 用来存元数据和需要低延迟访问的 I/O 管理结构，冷数据放 NVMe SSD，形成互补的分层存储架构。
