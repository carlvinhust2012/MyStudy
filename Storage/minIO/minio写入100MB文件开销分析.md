## 2. 存储空间成本（核心开销）

### 2.1 纠删码放大效应

MinIO 使用 Reed-Solomon 纠删码，原始数据被拆分为 **N 个数据分片 + M 个校验分片**，每块 1 MiB 数据生成 N+M 个分片。

**代码位置**: `erasure-coding.go:116-121`

```
ShardSize = ceil(blockSize / dataBlocks)
ShardFileSize = numBlocks × ShardSize   (+ bitrot hash 开销)
总存储 = ShardFileSize × (dataBlocks + parityBlocks)
```

## 3. Bitrot 哈希开销（微小但不可忽略）

### 3.1 哈希机制

MinIO 默认使用 **HighwayHash256S** (流式模式)，每个 shard 前面附加一个 **32 字节**的哈希值。

**代码位置**: `bitrot-streaming.go:57-65`, `xl-storage-format-v1.go:158`

```
磁盘上 part.1 的布局:

┌─────────┬──────────┬─────────┬──────────┬─────┬─────────┬──────────┐
│ Hash 0  │ Shard 0  │ Hash 1  │ Shard 1  │ ... │ Hash 99 │ Shard 99 │
│ 32 字节  │ 128 KiB  │ 32 字节  │ 128 KiB  │     │ 32 字节  │ 128 KiB  │
└─────────┴──────────┴─────────┴──────────┴─────┴─────────┴──────────┘
   ↑         ↑
   HighwayHash  数据分片

每个 Block 的 shard 前都有 32 字节哈希
```

### 3.2 哈希开销计算

```
以 8D+8P (ShardSize = 128 KiB) 为例:

每盘: 100 个 Block × 32 字节 = 3,200 字节 ≈ 3.1 KiB
全集群: 3,200 × 16 = 51,200 字节 ≈ 50 KiB

开销比例: 50 KiB / 200 MiB ≈ 0.024%
→ 微乎其微, 几乎可以忽略

以 12D+4P (ShardSize ≈ 85 KiB) 为例:

每盘: 100 × 32 = 3,200 字节
全集群: 3,200 × 16 = 51,200 字节 ≈ 50 KiB

开销比例: 50 KiB / 133 MiB ≈ 0.037%
→ 同样可以忽略
```

### 3.3 不同哈希算法的开销

```
┌──────────────────┬────────────┬──────────────┬───────────────────────────┐
│ 算法              │ 哈希大小    │ 模式         │ 100MB 文件每盘开销          │
├──────────────────┼────────────┼──────────────┼───────────────────────────┤
│ HighwayHash256S  │ 32 字节/块  │ 流式 (默认)  │ 100 × 32 = 3,200 字节     │
│ HighwayHash256   │ 32 字节    │ 整文件       │ 32 字节 (仅元数据)         │
│ SHA256           │ 32 字节    │ 整文件       │ 32 字节 (仅元数据)         │
│ BLAKE2b512       │ 64 字节    │ 整文件       │ 64 字节 (仅元数据)         │
└──────────────────┴────────────┴──────────────┴───────────────────────────┘

注: 整文件模式 (非默认) 的哈希存储在 xl.meta 的 ChecksumInfo 中
    流式模式 (默认) 的哈希内嵌在 part.1 数据流中
```

## 13. 关键结论

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        100MB 文件存储成本核心结论                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  1. 主要成本是纠删码存储放大                                              │
│     ├── 12D+4P (默认): 1.33x → 133 MB                                    │
│     ├── 8D+8P (高可用): 2.00x → 200 MB                                   │
│     └── 这占总成本的 99.9%+                                              │
│                                                                          │
│  2. 元数据开销几乎为零                                                    │
│     ├── Bitrot 哈希: 50 KB (0.024%)                                     │
│     ├── xl.meta: 6.3 KB (0.003%)                                        │
│     └── 合计不到 60 KB, 可忽略                                           │
│                                                                          │
│  3. CPU 和内存不是瓶颈                                                    │
│     ├── CPU: ~20-40 ms (SIMD 加速, 与 I/O 重叠)                         │
│     ├── 内存: ~3-4 MiB (流式处理, 与文件大小无关)                         │
│     └── 实际瓶颈是磁盘 I/O 和网络带宽                                    │
│                                                                          │
│  4. 写放大显著但可控                                                      │
│     ├── 12D+4P: 写入 133 MB (WA = 1.33x)                               │
│     ├── 8D+8P: 写入 200 MB (WA = 2.0x)                                 │
│     └── 读取无放大 (WA ≈ 1.0x)                                          │
│                                                                          │
│  5. 比 3 副本方案高效得多                                                  │
│     ├── MinIO 12D+4P: 1.33x (节省 56% vs 3副本)                         │
│     ├── HDFS/Ceph 3副本: 3.0x                                          │
│     └── 同时提供更强的容错能力 (4盘 vs 2节点)                             │
│                                                                          │
│  6. 小文件是效率杀手                                                      │
│     ├── 1 KB 文件: 元数据占比 22%                                        │
│     ├── 1 MiB 以上: 元数据可忽略                                         │
│     └── 建议合并小文件上传                                               │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```
