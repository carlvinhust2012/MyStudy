# MinIO 存储模型架构分析

## 1. 概述

MinIO 是一个**高性能、S3 兼容的对象存储系统**，采用 **Reed-Solomon 纠删码** 实现数据冗余和高可用。

**核心设计原则：**
- 无中心元数据服务器，**计算式路由**（一致性哈希）
- 数据冗余通过**纠删码**实现，非多副本
- **自愈合**：后台自动修复损坏数据

## 4. 纠删码机制

### 4.1 Reed-Solomon 编码

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     Reed-Solomon 纠删码原理                               │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  原始数据 (1 MiB Block)                                                  │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │                    D = D0 + D1 + D2 + D3                           │  │
│  │                    (4 个数据块, 每块 256 KiB)                      │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│                              │                                           │
│                              │ Reed-Solomon 编码                         │
│                              ▼                                           │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │                                                                    │  │
│  │   数据分片 (Data Shards)          校验分片 (Parity Shards)        │  │
│  │   ┌──────┐ ┌──────┐             ┌──────┐ ┌──────┐                │  │
│  │   │  D0  │ │  D1  │             │  P0  │ │  P1  │                │  │
│  │   │256KB │ │256KB │             │256KB │ │256KB │                │  │
│  │   └──────┘ └──────┘             └──────┘ └──────┘                │  │
│  │   ┌──────┐ ┌──────┐                                               │  │
│  │   │  D2  │ │  D3  │                                               │  │
│  │   │256KB │ │256KB │                                               │  │
│  │   └──────┘ └──────┘                                               │  │
│  │                                                                    │  │
│  │   总共: 4 Data + 2 Parity = 6 个分片 (写入 6 个磁盘)              │  │
│  │                                                                    │  │
│  └────────────────────────────────────────────────────────────────────┘  │
│                                                                          │
│  容错能力:                                                               │
│  ├── 丢失任意 2 个分片 (含校验) → 数据可恢复                             │
│  ├── 需要 4 个分片即可重建原始数据                                       │
│  └── 存储 overhead: 6/4 = 1.5x (50% 冗余)                              │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.3 读写仲裁

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        读写仲裁规则                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  示例: 4 Data + 2 Parity = 6 盘                                         │
│                                                                          │
│  写仲裁 (Write Quorum):                                                  │
│  ├── 需要至少 4 个盘写入成功                                              │
│  └── 当 data == parity 时，需要 data+1 = 5 个盘                          │
│                                                                          │
│  读仲裁 (Read Quorum):                                                   │
│  ├── 需要至少 4 个盘读取成功                                              │
│  └── 读取到的分片可以重建完整数据                                        │
│                                                                          │
│  容错分析:                                                               │
│                                                                          │
│  磁盘状态            可用盘    能否写    能否读                           │
│  ─────────────      ──────    ──────    ──────                           │
│  全部正常            6/6       ✅        ✅                               │
│  1 盘故障            5/6       ✅        ✅                               │
│  2 盘故障            4/6       ✅        ✅                               │
│  3 盘故障            3/6       ❌        ✅ (仍可读)                      │
│  4 盘故障            2/6       ❌        ❌                               │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.2 对象 → Set 路由

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     对象到 Set 的哈希路由                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  选定 Pool 后:                                                           │
│                                                                          │
│  objectName = "bucket/photos/img.jpg"                                   │
│  deploymentID = [16]byte{...}                                           │
│                                                                          │
│  setIndex = hashKey("SIPMOD+PARITY", objectName, setCount, deployID)    │
│                                                                          │
│  SIPMOD 算法:                                                            │
│  ├── 1. SIPHash(objectName) → 64bit hash                                │
│  ├── 2. hash % setCount → setIndex                                      │
│  └── 3. 加入 deploymentID 保证不同集群分布不同                           │
│                                                                          │
│  示例:                                                                   │
│  ├── setCount = 16 (16 个 Set)                                          │
│  ├── hash("img.jpg") = 0x...42                                          │
│  └── 0x42 % 16 = 2 → Set 2                                              │
│                                                                          │
│  特点:                                                                   │
│  ├── 确定性: 相同对象名始终路由到相同 Set                                 │
│  ├── 均匀性: 哈希均匀分布                                                │
│  └── 无状态: 不需要元数据服务器                                          │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

## 7. 数据写入流程

### 7.1 PUT Object 时序图

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant API as S3 API
    participant Pool as erasureServerPools
    participant Sets as erasureSets
    participant EO as erasureObjects
    participant EC as Erasure Coder
    participant Disk0 as Disk 0
    participant Disk1 as Disk 1
    participant DiskN as Disk N

    Client->>API: 1. PUT object (数据流)

    API->>Pool: 2. PutObject(bucket, object, reader)

    Pool->>Pool: 3. 选择可用 Pool (加权随机)

    Pool->>Sets: 4. 路由到 erasureSets

    Sets->>Sets: 5. hashKey(objectName) → Set Index

    Sets->>EO: 6. 委托给 erasureObjects

    EO->>EO: 7. 获取分布式锁 (dsync)

    EO->>EO: 8. 确定 parity 数量
    Note over EO: 检查在线磁盘数，动态升级

    EO->>EO: 9. 创建 FileInfo
    Note over EO: 分配 DataDir (UUID)

    loop 每 1 MiB Block
        EO->>EC: 10. Encode(block)
        EC->>EC: 11. 分割为 Data + Parity 分片

        par 并行写入
            EC->>Disk0: 12a. 写分片 0
            EC->>Disk1: 12b. 写分片 1
            EC->>DiskN: 12c. 写分片 N
        end

        Note over EC: 等待 Quorum 成功
    end

    par 写元数据
        EO->>Disk0: 13a. renameData + xl.meta
        EO->>Disk1: 13b. renameData + xl.meta
        EO->>DiskN: 13c. renameData + xl.meta
    end

    Note over EO: 原子提交 (Quorum 成功)

    EO-->>API: 14. 返回成功
    API-->>Client: 15. 200 OK

    Note over EO: 如有磁盘离线，加入 MRF 治愈队列
```

## 13. 总结

| 问题 | 答案 |
|------|------|
| **存储模型** | Pool → Set → Disk 三层结构 |
| **冗余机制** | Reed-Solomon 纠删码 (非多副本) |
| **路由方式** | SIPMOD 哈希 (无元数据服务器) |
| **数据块大小** | 1 MiB (blockSizeV2) |
| **元数据存储** | xl.meta (MessagePack 格式) |
| **数据完整性** | HighwayHash (Bitrot 保护) |
| **容错能力** | 丢失 N/2 个分片仍可读 |
| **自愈合** | 后台自动修复损坏数据 |
| **一致性** | Quorum 读写 (多数派) |

### 核心设计理念

1. **计算代替元数据**：对象位置通过哈希计算，无需元数据服务器
2. **纠删码优于多副本**：更低的存储开销，更高的耐久性
3. **自底向上的自愈**：从读取触发到后台扫描，层层修复
4. **无共享架构**：每个节点对等，无单点故障
