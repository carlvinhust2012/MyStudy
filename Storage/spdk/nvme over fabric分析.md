# NVMe over Fabrics 与 RDMA 网络技术分析

## 1. 一句话概括

NVMe over Fabrics（NVMe-oF）是把本地 NVMe 协议搬到网络上，让远程服务器像访问本地盘一样访问远端 SSD；其中 "Fabrics" 指的是数据中心网络结构（Fabric），RDMA 是最主流的一种 Fabric 传输方式。

## 2. NVMe-oF 解决什么问题

```
传统方案（以 iSCSI 为例）:
  ┌────────┐     ┌────────┐     ┌────────┐
  │ 应用   │ → SCSI │ → TCP │ → 远端SSD │
  └────────┘     └────────┘     └────────┘
  问题: SCSI 是为机械硬盘设计的协议，效率低，队列深度只有 32

NVMe-oF 方案:
  ┌────────┐     ┌─────────┐     ┌────────┐
  │ 应用   │ → NVMe │ → Fabric │ → 远端SSD │
  └────────┘     └─────────┘     └────────┘
  优势: 直接复用 NVMe 协议，去掉 SCSI 转换层，延迟更低
```

### 2.1 本地 NVMe vs NVMe-oF vs iSCSI

| 对比项 | 本地 NVMe | NVMe over Fabrics | iSCSI |
|---|---|---|---|
| 协议 | NVMe/PCIe | NVMe/Fabric | SCSI/TCP |
| 延迟 | ~20us | ~10-100us（取决于网络） | ~100-500us |
| 带宽 | ~7 GB/s | ~3-25 GB/s（取决于网络带宽） | ~1 GB/s（10GbE） |
| SSD 距离 | 机箱内 | 数据中心内甚至跨机房 | 无限制 |
| 队列深度 | 65536 | 65536 | 32 |
| CPU 开销 | 低（DMA） | RDMA 模式低，TCP 模式中等 | 高（内核协议栈） |

## 3. "Fabrics" 是什么意思

### 3.1 Fabric 的含义

**Fabric 就是"数据中心网络结构"的统称**，不是某个具体协议，而是指交换机、网卡、线缆、协议组合而成的整体网络架构。

```
Fabric = 数据中心网络结构
       = 交换机 + 网卡 + 线缆 + 协议
       = 让多台服务器互相通信的基础设施

所以:
  NVMe over Fabrics = "NVMe 协议跑在各种网络结构上"
  而不是 "NVMe over Network"（Network 太笼统）
  Fabric 强调的是数据中心级别的网络架构
```

### 3.2 为什么叫 Fabrics 不叫 Network

```
Network:  泛指所有网络（包括互联网、局域网、WiFi 等）
Fabric:   专指数据中心内部的高速互联网络结构
          强调的是"织成一张网"的结构感
          常用于存储网络（SAN Fabric）、集群互联

类比:
  Network ≈ "公路网"（泛指）
  Fabric  ≈ "高速公路网"（专指高速互联的结构化网络）
```

## 4. NVMe-oF 支持的三种传输方式

NVMe-oF 定义了 NVMe 协议怎么跑在不同的网络 Fabric 上:

### 4.1 NVMe over RDMA（最主流）

```
┌───────┐   ┌───────┐   ┌────────────┐   ┌──────┐
│ 应用  │ → │ NVMe  │ → │ RDMA 网络   │ → │ SSD  │
└───────┘   └───────┘   └────────────┘   └──────┘

特点:
  - 绕过内核，CPU 零拷贝
  - 延迟最低（~10-30us）
  - 需要特殊网卡（RoCE 或 iWARP）

RDMA 底层链路技术:
  - RoCE（RDMA over Converged Ethernet）: 最流行，跑在以太网上
  - iWARP（Internet Wide Area RDMA Protocol）: 跑在标准 TCP/IP 上
  - InfiniBand: 专门的 RDMA 网络，性能最好但贵
```

### 4.2 NVMe over TCP（最新，最易部署）

```
┌───────┐   ┌───────┐   ┌────────────┐   ┌──────┐
│ 应用  │ → │ NVMe  │ → │ TCP/IP 网络 │ → │ SSD  │
└───────┘   └───────┘   └────────────┘   └──────┘

特点:
  - 走普通 TCP/IP 以太网，无需特殊硬件
  - 部署成本最低，利用现有网络基础设施
  - 需要经过内核协议栈，有数据拷贝开销
  - 延迟: ~50-100us
  - NVMe 1.4 标准正式引入
```

### 4.3 NVMe over FC（传统企业存储）

```
┌───────┐   ┌───────┐   ┌────────────┐   ┌──────┐
│ 应用  │ → │ NVMe  │ → │ FC 网络    │ → │ SSD  │
└───────┘   └───────┘   └────────────┘   └──────┘

特点:
  - 延续企业存储的 FC（Fibre Channel）基础设施
  - 已有 FC SAN 的企业可以平滑升级
  - 延迟: ~10-50us
```

### 4.4 三种传输方式对比

| 对比项 | NVMe over RDMA | NVMe over TCP | NVMe over FC |
|---|---|---|---|
| 延迟 | 最低（~10-30us） | 中等（~50-100us） | 低（~10-50us） |
| 部署难度 | 高（需 RoCE/iWARP 网卡） | 低（普通以太网） | 中（需 FC 交换机） |
| 硬件成本 | 高 | 低 | 高 |
| CPU 开销 | 最低（零拷贝） | 中等（内核拷贝） | 低 |
| 普及程度 | 最主流 | 快速增长 | 传统企业 |
| 典型用户 | 云厂商、HPC、数据库 | 中小企业、云原生 | 金融、传统企业 |

## 5. RDMA 为什么快

RDMA（Remote Direct Memory Access）是 NVMe-oF 性能最优的关键:

### 5.1 普通 TCP vs RDMA 传输路径

```
普通 TCP/IP 传输（内核参与，拷贝多）:

  发送端:
    应用缓冲区 → 内核缓冲区（拷贝1）→ 网卡（拷贝2）→ 网络
    每次经过内核: 上下文切换 + 中断处理

  接收端:
    网络 → 网卡 → 内核缓冲区（拷贝3）→ 应用缓冲区（拷贝4）
    每次经过内核: 上下文切换 + 中断处理

  总共: 4 次数据拷贝 + 2 次上下文切换 + 2 次中断
  延迟: ~50us+

RDMA 传输（绕过内核，零拷贝）:

  发送端:
    应用缓冲区 → 网卡 → 网络（直接从用户态内存读取，不经过内核）

  接收端:
    网络 → 网卡 → 应用缓冲区（直接写入用户态内存，不经过内核）

  总共: 0 次数据拷贝 + 0 次上下文切换 + 0 次中断
  延迟: ~1-2us（纯网络传输）
```

### 5.2 RDMA 三种底层链路技术

```
┌──────────────────────────────────────────────────────────┐
│                  RDMA 三种底层链路                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  InfiniBand（IB）                                         │
│  - 专门的 RDMA 网络，不是以太网                              │
│  - 性能最好（延迟 ~1us，带宽 200Gbps+）                     │
│  - 最贵，需要专用交换机和网卡                                 │
│  - HPC 超算场景常用                                        │
│                                                          │
│  RoCE（RDMA over Converged Ethernet）                      │
│  - 跑在普通以太网上，用 lossless 以太网                      │
│  - 性能接近 InfiniBand                                     │
│  - 只需换网卡，不需要换交换机（需 PFC 配置）                  │
│  - 最主流的 RDMA 方案                                       │
│  - RoCE v1: 同子网                                         │
│  - RoCE v2: 支持跨子网路由（走 UDP）                         │
│                                                          │
│  iWARP（Internet Wide Area RDMA Protocol）                 │
│  - 跑在标准 TCP/IP 上                                      │
│  - 不需要 lossless 以太网                                   │
│  - 可以跨广域网                                             │
│  - 性能低于 RoCE，但部署简单                                 │
│  - 使用较少，主要在广域 RDMA 场景                             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 5.3 三种链路技术对比

| 对比项 | InfiniBand | RoCE v2 | iWARP |
|---|---|---|---|
| 链路层 | IB 网络 | 以太网（UDP） | 以太网（TCP） |
| 延迟 | ~1us | ~2-5us | ~5-10us |
| 需要 lossless | 本身就是 | 需要（PFC） | 不需要 |
| 跨子网 | 支持 | v2 支持 | 支持 |
| 部署成本 | 最贵 | 中等（换网卡） | 最低 |
| 性能 | 最高 | 接近 IB | 最低 |
| 生态 | HPC 专用 | 数据中心主流 | 较少使用 |

## 6. NVMe-oF 典型架构

```
┌─────────────────────────────────────────────────────────┐
│                   NVMe-oF 存储架构                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ 计算节点1 │  │ 计算节点2 │  │ 计算节点3 │              │
│  │ NVMe     │  │ NVMe     │  │ NVMe     │              │
│  │ Initiator│  │ Initiator│  │ Initiator│              │
│  └─────┬────┘  └─────┬────┘  └─────┬────┘              │
│        │              │              │                   │
│        └──────────────┼──────────────┘                   │
│                  ┌─────┴─────┐                           │
│                  │  网络 Fabric│  (RoCE / TCP / FC)      │
│                  │  交换机    │                           │
│                  └─────┬─────┘                           │
│                        │                                 │
│        ┌───────────────┼───────────────┐                 │
│        │               │               │                 │
│  ┌─────┴────┐  ┌──────┴─────┐  ┌──────┴─────┐          │
│  │ 存储节点1 │  │ 存储节点2  │  │ 存储节点3  │          │
│  │ NVMe     │  │ NVMe      │  │ NVMe      │          │
│  │ Target   │  │ Target    │  │ Target    │          │
│  │ + 本地SSD│  │ + 本地SSD  │  │ + 本地SSD  │          │
│  └──────────┘  └──────────┘  └──────────┘          │
│                                                         │
│  关键术语:                                                │
│  - Initiator: 发起 NVMe 请求的客户端（计算节点）              │
│  - Target:    处理 NVMe 请求的服务端（存储节点）               │
│  - SPDK:      常用的用户态 NVMe Target 开源框架               │
│  - subsystem: Target 上的存储命名空间，类似 LUN               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 7. 应用场景

| 场景 | 推荐传输方式 | 原因 |
|---|---|---|
| 数据库（MySQL、TiDB） | NVMe over RDMA（RoCE） | 低延迟、高 IOPS |
| 高性能计算（HPC） | NVMe over RDMA（InfiniBand） | 极致低延迟 |
| 虚拟化平台 | NVMe over TCP 或 RDMA | 灵活，成本低 |
| 容器/K8s 持久卷 | NVMe over TCP | 部署简单，兼容性好 |
| 企业存储阵列 | NVMe over FC | 延续 FC 基础设施 |
| 存储资源池化 | NVMe over RDMA | 高吞吐，低延迟 |

## 8. SPDK 与 NVMe-oF 的关系

SPDK（Storage Performance Development Kit）是 NVMe-oF 生态中最核心的开源软件:

```
SPDK 提供的 NVMe-oF 组件:

  ┌─────────────────────────────────────────────┐
  │                  SPDK                        │
  ├─────────────────────────────────────────────┤
  │                                             │
  │  nvmf_tgt（NVMe-oF Target）                  │
  │  - 用户态 NVMe Target 实现                    │
  │  - 支持 RDMA / TCP / FC                      │
  │  - 零拷贝，轮询模式，无中断                    │
  │                                             │
  │  nvme（NVMe Driver）                         │
  │  - 用户态 NVMe 驱动                          │
  │  - 直接操作 PCIe，绕过内核                     │
  │                                             │
  │  bdev（Block Device Layer）                  │
  │  - 统一的块设备抽象层                         │
  │  - 支持后端: NVMe / Ceph / RAID / 内存等     │
  │                                             │
  └─────────────────────────────────────────────┘

SPDK 的核心优势: 全用户态
  - 绕过内核，无系统调用开销
  - 轮询模式（PMD），无中断开销
  - 零拷贝，无数据拷贝开销
  - 与 DPDK 网络栈配合实现端到端零拷贝
```

## 9. 与 DAOS 的关系

DAOS（前面分析过的分布式存储系统）就大量使用了 NVMe-oF 技术:

```
DAOS 存储节点:

  客户端 → NVMe-oF（RDMA/RoCE）→ 服务端 Target
                                     │
                                     ├── SCM（VOS 元数据）
                                     └── NVMe SSD（大对象数据）

DAOS 使用 SPDK 的 nvmf_tgt 作为存储 Target
客户端通过 libfabric（OFI）访问 RDMA 网络
端到端路径: 客户端 NVMe-oF → RDMA → 服务端 SPDK → NVMe SSD
```

## 10. 总结

```
NVMe-oF = NVMe 协议 + 网络传输（Fabric）
  目标: 让远程 SSD 访问接近本地 NVMe 的体验

Fabric = 数据中心网络结构
  包括: RDMA 网络 / TCP/IP 网络 / FC 网络
  RDMA 是性能最优的 Fabric 传输方式

RDMA = Remote Direct Memory Access
  核心: 绕过内核，零拷贝，CPU 不参与数据搬运
  底层: InfiniBand / RoCE（以太网） / iWARP（TCP）

SPDK = NVMe-oF 的开源软件实现
  核心: 全用户态，零拷贝 + 轮询模式 + 无中断
  是 DAOS 等分布式存储的底层存储框架
```
