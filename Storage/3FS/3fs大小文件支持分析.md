# 3FS 大文件与小文件支持分析

## 一、核心结论

**3FS 为大文件（GB~TB 级）高吞吐场景优化，对大量小文件（< chunkSize）支持有限。**

| 场景 | 友好度 | 说明 |
|------|--------|------|
| 100 GB+ Checkpoint 保存/加载 | 极好 | 条带化 + RDMA + 批量 I/O 全部生效 |
| 1~10 GB 模型权重 | 好 | 条带化并行度足够 |
| 1~100 MB 训练样本文件 | 中等 | 元数据开销占比增大 |
| <512 KB 小文件 | 差 | Chunk 空间浪费、元数据瓶颈、无法并行化 |

---

## 二、小文件的空间浪费

### 2.1 最小 Chunk 分配

```
假设 chunkSize = 4 MB:

  1 KB 文件 → 分配 1 个 4MB Chunk → 空间利用率 0.025%
  10 KB 文件 → 分配 1 个 4MB Chunk → 空间利用率 0.25%
  1 MB 文件 → 分配 1 个 4MB Chunk → 空间利用率 25%
  4 MB 文件 → 分配 1 个 4MB Chunk → 空间利用率 100%
  100 GB 文件 → 分配 25600 个 Chunk → 空间利用率 ~100%
```

### 2.2 3 副本放大

```
1 KB 文件 × 3 副本 × 4 MB Chunk:
  实际磁盘占用 = 12 MB
  有效数据 = 1 KB
  浪费比例 = 99.99%

100 万个 1KB 小文件:
  有效数据 = 1 GB
  实际占用 = 12 TB
```

### 2.3 ChunkSize 与空间浪费的关系

| chunkSize | 1KB 文件利用率 | 100KB 文件利用率 | 1MB 文件利用率 |
|-----------|-------------|---------------|-------------|
| 512 KB | 0.2% | 20% | 100% |
| 1 MB | 0.1% | 10% | 100% |
| 4 MB | 0.025% | 2.5% | 25% |
| 16 MB | 0.006% | 0.6% | 6.25% |

## 三、小文件的元数据瓶颈

### 3.1 每文件的元数据开销

```
每个小文件在 FDB 中的存储:

  INOD + inodeId → InodeData     (~200 bytes FDB value)
  DENT + parent + name → DirEntryData  (~100 bytes FDB value)

  FDB 内部存储开销: Key + Value + 索引 ≈ 300~500 bytes/文件
```

### 3.2 每文件的 RPC 开销

```
写一个 1KB 小文件: 4 次 RPC

  ① Meta: CreateReq   (创建 Inode + DENT + Session) → 1 次 FDB 事务
  ② Storage: WriteReq  (写 1KB 数据, 链式复制 3 跳)    → 1 次数据 RPC
  ③ Meta: SyncReq     (更新文件长度)                    → 1 次 FDB 事务
  ④ Meta: CloseReq    (关闭文件, 删除 Session)          → 1 次 FDB 事务

读一个 1KB 小文件: 5 次 RPC

  ① Meta: 路径解析    (N 层 DENT snapshotGet)          → N 次 FDB 读取
  ② Meta: GetAttrReq (获取 Inode 信息)                 → 1 次 FDB get
  ③ Storage: ReadReq  (读 1KB 数据)                    → 1 次数据 RPC

  总计: 4~5 次 RPC, 其中 3~4 次是 Meta RPC
  实际数据传输: 1 KB
  控制开销远大于数据
```

### 3.3 元数据 vs 数据的比例

```
大文件 (100 GB):
  元数据: ~500 bytes
  数据: 100 GB
  比例: 1 : 2 亿
  元数据可忽略 ✓

小文件 (1 KB):
  元数据: ~500 bytes
  数据: 1 KB
  比例: 1 : 2
  元数据占比 50% ✗

100 万个小文件 (1 KB 每个):
  元数据: ~500 MB (FDB)
  数据: ~1 GB (磁盘)
  元数据占总存储的 33%
  Meta RPC 成为绝对瓶颈
```

## 四、小文件无法利用 3FS 核心优化

### 4.1 条带化失效

```
大文件: 500 MB 写入, chunkSize=4MB, stripeSize=4

  Chunk 0 → Chain 0
  Chunk 1 → Chain 1    4 条链并行
  Chunk 2 → Chain 2    聚合带宽
  Chunk 3 → Chain 3

小文件: 1 KB 写入

  Chunk 0 → Chain 0    只有 1 个 Chunk
  其他 3 条链空闲      无法并行
```

### 4.2 批量 I/O 失效

```
batchWrite 参数:
  max_batch_size: 128 个 IO / 批次
  max_batch_bytes: 4 MB / 批次

大文件 500 MB 写入:
  500 MB / 4 MB = 125 个 IO → 1 个批次即可
  或跨多个节点并行: 4 个批次 × 128 IO = 512 IO 并发

小文件 1 KB 写入:
  只有 1 个 IO → 批量化完全无用
```

### 4.3 RDMA 零拷贝收益极低

```
RDMA 传输开销:
  控制开销 (setup + completion): ~10 μs
  数据传输: 1 KB / 100 Gbps ≈ 0.08 μs

大文件 4 MB:
  数据传输: 4 MB / 100 Gbps ≈ 320 μs
  控制开销占比: 10 / 330 ≈ 3% ✓

小文件 1 KB:
  数据传输: 0.08 μs
  控制开销占比: 10 / 10.08 ≈ 99.2% ✗
  RDMA 的优势完全被控制开销淹没
```

## 五、3FS 缺失的小文件优化

| 优化技术 | 说明 | 3FS 是否支持 |
|---------|------|-------------|
| Tail Packing | 小文件附加到前一个 Chunk 的空闲尾部 | 否 |
| 小文件合并 | 多个小文件打包到同一个 Chunk | 否 |
| 文件级 Compaction | 将多个小 Chunk 合并为大 Chunk | 否 |
| 小文件缓存 | 缓存热点小文件的 Inode 和数据 | 部分 (Inode 缓存有, 数据缓存无) |
| 列存格式 | 多行共享元数据 (Doris 模式) | 否 (POSIX 文件系统, 非列存) |
| 对象聚合 | 多文件打包为单个对象 (如 S3 multipart) | 否 |

### 为什么不做

```
设计取舍:

  加入小文件优化的代价:
  ├── Tail Packing → Chunk 读写路径复杂化, 需要额外的尾部空间索引
  ├── 小文件合并 → 打破 "一个文件 = 一个 Chunk 区间" 的简洁映射
  ├── 文件级 Compaction → 与现有 GC 机制冲突, 增加写放大
  └── 每个优化都增加代码复杂度, 可能影响大文件的核心性能

  3FS 的定位:
    "为 AI 大文件高吞吐做极致优化,
     而非做一个所有场景都 80 分的通用文件系统"

  小文件的正确做法:
    上层聚合 → tar/zip 打包 → 作为大文件存储 → 按需解压
```

## 六、大文件 vs 小文件完整对比

| 维度 | 1 KB 小文件 | 100 GB 大文件 | 倍数差距 |
|------|----------|-----------|---------|
| **空间利用率** | 0.025% (4MB Chunk) | ~100% | 4000x |
| **元数据/数据比** | 500B : 1KB | 500B : 100GB | 1 亿倍 |
| **写 Meta RPC** | 3 次 | 3 次 (相同) | 1x |
| **写 Storage RPC** | 1 次 | 25600 次 (多链并行) | — |
| **写吞吐上界** | ~1 MB/s (RPC 延迟限制) | ~100 GB/s (带宽限制) | 100000x |
| **读 Meta RPC** | 3~4 次 | 3~4 次 (相同) | 1x |
| **读吞吐上界** | ~1 MB/s | ~100 GB/s | 100000x |
| **条带化并行度** | 1 Chunk, 1 Chain | 25600 Chunks, N Chains | 25600x |
| **batchWrite 并发** | 1 IO | 128 IO/批次 × 多节点 | 128x+ |
| **RDMA 效率** | 控制开销占 99% | 控制开销占 3% | 33x |

## 七、AI 场景的文件大小分布

```
典型 AI 训练集群的文件大小分布:

  Checkpoint (权重):     10~1000 GB / 文件    → 3FS 最优场景 ✓
  模型权重 (推理):       1~100 GB / 文件      → 3FS 优势场景 ✓
  训练样本 (图片/文本):  100 KB~100 MB / 文件 → 3FS 中等, 建议打包
  Tokenizer/Config:     <100 KB / 文件       → 3FS 弱项, 数量少可接受
  日志文件:             KB~MB 级, 高频写入    → 3FS 弱项, 建议用其他系统

结论: AI 核心工作负载 (Checkpoint + 权重) 都是大文件
      小文件数量少, 不构成系统瓶颈
      3FS 的设计取舍是合理的
```
