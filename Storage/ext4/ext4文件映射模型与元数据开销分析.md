# Ext4 文件映射模型与元数据开销分析

## 1. 一句话概括

Ext4 中每个文件对应 1 个 inode，inode 通过 Extent（连续物理块范围）映射到磁盘上的数据块；1 亿个文件的纯元数据开销约 27GB（占磁盘 1.7%），查找速度取决于单目录文件数而非文件总数。

## 2. 文件到磁盘的映射模型

### 2.1 完整映射链路

```
文件 file.txt (1MB)
  │
  ├── 1 个 Inode（256 字节）
  │     - 文件类型、大小、权限、时间戳
  │     - i_block[15]: 存储 Extent 或 Extent Tree 指针
  │
  └── Extent 映射 → 物理块
        Extent 1: 逻辑块 0-255 → 物理块 1000-1255（连续 1MB）
        仅需 12 字节描述
```

### 2.2 Extent vs 传统间接指针

```
ext2/3 间接指针方式:
  ┌──────────────────────────────────────────────────┐
  │ Inode i_block[15]                                │
  │ ┌──────┬──────┬────────────────┬──────────────┐  │
  │ │0-11  │ 12   │ 13            │ 14            │  │
  │ │直接  │间接  │二级间接        │三级间接        │  │
  │ │指针  │指针  │指针           │指针           │  │
  │ │12块  │256块 │256x256块      │256x256x256块  │  │
  │ └──────┴──────┴────────────────┴──────────────┘  │
  │                                                    │
  │  1MB 文件（256 个块）需要:                          │
  │  - 12 个直接块指针（48KB）                          │
  │  - 1 个间接块指针 → 1 个间接块（剩余 244 块）       │
  │  - 额外读 1 个间接块来获取物理块号                   │
  └──────────────────────────────────────────────────┘

ext4 Extent 方式:
  ┌──────────────────────────────────────────────────┐
  │ Inode i_block[15]                                │
  │ ┌──────────────┬──────────────┬──────────────┐   │
  │ │ Extent Header│ Extent 1     │ Extent 2 ...  │   │
  │ │ 12 字节      │ 12 字节      │ 12 字节      │   │
  │ └──────────────┴──────────────┴──────────────┘   │
  │                                                    │
  │  1MB 文件（256 个连续块）:                          │
  │  - 1 个 Extent: ee_block=0, ee_len=256,            │
  │                ee_start=1000                        │
  │  - 仅 12 字节，不需要间接块                         │
  │  - 一次查找就得到所有物理块位置                      │
  └──────────────────────────────────────────────────┘
```

### 2.3 Extent 结构详解

```
单个 Extent（12 字节）:
  ┌──────────────────┬──────────────────┬──────────────────┐
  │ ee_block (32bit) │ ee_len (16bit)   │ ee_start (32bit) │
  │ 起始逻辑块号     │ 连续块数         │ 起始物理块号     │
  └──────────────────┴──────────────────┴──────────────────┘

  最大描述范围: 32768 个连续块 = 128MB（4KB 块大小）
  Inode 中最多直接存 4 个 Extent（60B / 12B = 4 个，其余给 Header）

连续分配时（最优情况）:
  1GB 文件 → 8 个 Extent（每个 128MB）→ 96 字节

碎片化时（需要 Extent Tree）:
  1GB 文件，碎片严重 → 可能几百个 Extent
  → 自动升级为 Extent Tree（B-Tree 索引）

Extent Tree:
  ┌─────────────────────────────┐
  │ Root（在 inode 的 i_block 中）│
  │ Extent Header + Index 1-4   │
  │ 指向子节点                   │
  ├─────────────────────────────┤
  │ Internal Node               │
  │ Index 指向更深层节点          │
  ├─────────────────────────────┤
  │ Leaf Node（在数据块中）       │
  │ 实际的 Extent（物理块映射）   │
  └─────────────────────────────┘
  查找深度: O(log N)，N 为 Extent 数量
```

### 2.4 逻辑块号到物理块号的转换流程

```
读取 file.txt 的第 100KB（偏移 102400 字节，4KB 块）:

  Step 1: 计算逻辑块号
    logical_block = 102400 / 4096 = 25

  Step 2: 在 inode 中查找 Extent
    遍历 Extent 列表:
    Extent 1: ee_block=0, ee_len=256, ee_start=1000
    判断: 0 <= 25 < 256 → 命中

  Step 3: 计算物理块号
    physical_block = ee_start + (logical_block - ee_block)
    physical_block = 1000 + (25 - 0) = 1025

  Step 4: 读取物理块 1025

  如果使用 Extent Tree:
    Step 2 变为 B-Tree 查找
    从根节点开始，比较逻辑块号范围
    O(log N) 次磁盘 I/O（N 为 Extent 数量）
```

## 3. 元数据开销分析

### 3.1 Superblock（超级块）详解

```
  ┌──────────────────────────────────────────────────────────┐
  │ Superblock                                               │
  ├──────────────────────────────────────────────────────────┤
  │                                                          │
  │  位置: Block Group 0 的第 0 个块（1024 字节偏移处）        │
  │  大小: 1024 字节（固定）                                  │
  │  数量: 整个文件系统只有 1 个（在 BG 0 中）                 │
  │  备份: 在 BG 1, 3, 5, 7, 9...（每 2^N 个 BG）有备份      │
  │                                                          │
  │  作用: 记录整个文件系统的全局参数                          │
  │        是挂载文件系统时内核读取的第一个结构                  │
  │        如果 Superblock 损坏，文件系统无法挂载               │
  │                                                          │
  │  关键字段:                                                │
  │  ┌───────────────────────────┬─────────────────────────┐ │
  │  │ 字段                      │ 说明                     │ │
  │  ├───────────────────────────┼─────────────────────────┤ │
  │  │ s_inodes_count            │ Inode 总数               │ │
  │  │ s_blocks_count_lo         │ 总块数（低 32 位）        │ │
  │  │ s_log_block_size          │ 块大小 2^(10+x)          │ │
  │  │                          │ 0=1KB 1=2KB 2=4KB        │ │
  │  │ s_blocks_per_group        │ 每 BG 的块数             │ │
  │  │ s_inodes_per_group        │ 每 BG 的 Inode 数        │ │
  │  │ s_inode_size              │ Inode 大小（256 字节）    │ │
  │  │ s_first_data_block        │ 第一个数据块号（0 或 1）   │ │
  │  │ s_journal_inum            │ 日志文件的 Inode 号       │ │
  │  │ s_feature_compat          │ 兼容特性标志位            │ │
  │  │ s_feature_incompat        │ 不兼容特性标志位          │ │
  │  │ s_feature_ro_compat       │ 只读兼容特性标志位        │ │
  │  │ s_uuid                    │ 文件系统唯一标识（128位）  │ │
  │  │ s_volume_name             │ 卷标名称                  │ │
  │  │ s_last_mount_time         │ 最后挂载时间              │ │
  │  │ s_last_write_time         │ 最后写入时间              │ │
  │  │ s_mmp_block               │ 多挂载保护块号            │ │
  │  │ s_desc_size               │ GDT 描述符大小（32/64B）  │ │
  │  └───────────────────────────┴─────────────────────────┘ │
  │                                                          │
  │  mount 流程中 Superblock 的作用:                          │
  │  ┌──────────────────────────────────────────────┐       │
  │  │ mount /dev/sda1 /mnt                         │       │
  │  │   → 内核读 BG 0 的 Superblock                 │       │
  │  │   → 检查 s_magic == 0xEF53（ext4 标识）       │       │
  │  │   → 检查 s_feature_incompat 是否有内核不支持   │       │
  │  │   → 从 s_log_block_size 获取块大小             │       │
  │  │   → 从 s_inodes_per_group 计算布局             │       │
  │  │   → 读取 GDT → 定位所有 BG 的元数据位置        │       │
  │  │   → 挂载完成                                   │       │
  │  └──────────────────────────────────────────────┘       │
  │                                                          │
  │  为什么需要备份:                                           │
  │    Superblock 是文件系统的"钥匙"                           │
  │    BG 0 的 Superblock 损坏 → 用备份恢复                    │
  │    备份间隔: BG 0, 1, 3, 5, 7, 9, 25, 27, 49...          │
  │    （sparse_super 特性控制备份位置）                       │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

### 3.2 每个文件的元数据组成

```
1 个文件需要:

  ┌──────────────────────────────────────────────────────┐
  │ 元数据组成                                            │
  ├──────────────────────────┬───────────────────────────┤
  │ Inode                    │ 256 字节（固定）           │
  │ 目录项                    │ ~16 字节（平均文件名8字节）│
  │ Block Bitmap 中的标记位   │ 1 bit                    │
  │ Inode Bitmap 中的标记位   │ 1 bit                    │
  ├──────────────────────────┼───────────────────────────┤
  │ 合计                     │ ~272 字节/文件            │
  └──────────────────────────┴───────────────────────────┘
```

### 3.3 Inode Bitmap 与 Block Bitmap 详解

```
每个 Block Group 有两个位图，各占 1 个块（4KB），各管理 32768 个对象:

  ┌──────────────────────────────────────────────────────────┐
  │ Inode Bitmap（Inode 位图）                                │
  ├──────────────────────────────────────────────────────────┤
  │                                                          │
  │  作用: 记录该 BG 中每个 Inode 的分配状态                    │
  │  大小: 1 个块（4KB）= 32768 个 bit                        │
  │  每个 bit 对应 1 个 Inode                                 │
  │                                                          │
  │  Bit 值含义:                                              │
  │    0 = 空闲（该 Inode 未被分配）                           │
  │    1 = 已使用（该 Inode 已分配给某个文件/目录）              │
  │                                                          │
  │  示例（前 20 个 bit）:                                    │
  │    11001100 11111111 11111111 11000000                   │
  │    ^^^^^^^^                                              │
  │    bit0=1: inode 已用（通常是丢失+Found目录）              │
  │    bit1=1: inode 已用（当前目录 "."）                      │
  │    bit2=0: inode 空闲                                    │
  │    bit3=0: inode 空闲                                    │
  │    bit4=1: inode 已用（父目录 ".."）                      │
  │    bit5=1: inode 已用（某个文件）                         │
  │    bit6=0: inode 空闲                                    │
  │    bit7=0: inode 空闲                                    │
  │    bit8-15: 全部已用（预留给系统 inode）                   │
  │                                                          │
  │  使用场景:                                                │
  │    创建文件 → 搜索 Inode Bitmap 找到第一个 0 bit            │
  │    删除文件 → 将对应 bit 置为 0                            │
  │    格式化 → 所有 bit 置为 0                                │
  │    fsck  → 遍历所有 bit，检查 inode 是否一致               │
  │                                                          │
  └──────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────┐
  │ Block Bitmap（数据块位图）                                 │
  ├──────────────────────────────────────────────────────────┤
  │                                                          │
  │  作用: 记录该 BG 中每个数据块的分配状态                     │
  │  大小: 1 个块（4KB）= 32768 个 bit                        │
  │  每个 bit 对应 1 个 4KB 的数据块                          │
  │                                                          │
  │  Bit 值含义:                                              │
  │    0 = 空闲（该块未被分配）                               │
  │    1 = 已使用（该块已分配给某个文件或元数据）               │
  │                                                          │
  │  注意: Block Bitmap 管理的是整个 BG 的所有块，              │
  │       包括 Inode Table、目录块、数据块等                    │
  │       但 Inode Table 占用的块在创建 FS 时就标记为 1         │
  │                                                          │
  │  使用场景:                                                │
  │    写文件 → 搜索 Block Bitmap 找连续的 0 bit（延迟分配时）  │
  │    删除文件 → 将对应数据块的 bit 置为 0                    │
  │    mballoc → 找连续 N 个 0 bit 一次性分配                  │
  │    fsck  → 遍历所有 bit，检查块是否被正确引用               │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

### 3.4 1 亿文件的总元数据开销

```
┌──────────────────────────────────────────────────────────┐
│                  1 亿文件的元数据开销                       │
├───────────────────────┬────────────────┬──────────────────┤
│ 项目                   │ 计算            │ 大小             │
├───────────────────────┼────────────────┼──────────────────┤
│ Inode 表              │ 1亿 x 256B     │ 25.6 GB         │
│ 目录项                │ 1亿 x 16B      │ 1.6 GB          │
│ Inode Bitmap          │ 1亿 bits       │ 12.5 MB         │
│ Block Bitmap          │ 取决于数据量    │ 略              │
│ 块组描述符表（GDT）    │ ~12208 BG      │ ~0.8 MB         │
│ 超级块                │ 1 个           │ 1 KB            │
├───────────────────────┼────────────────┼──────────────────┤
│ 纯元数据合计           │                │ 约 27.2 GB      │
├───────────────────────┼────────────────┼──────────────────┤
│ 支撑这些文件的最小磁盘  │                │ 约 1.56 TB      │
│ 元数据占磁盘比例       │                │ 约 1.7%         │
└───────────────────────┴────────────────┴──────────────────┘
```

### 3.5 空间计算推导

```
块组参数（4KB 块大小，默认配置）:

  每个块组:
    块数 = 32768
    大小 = 32768 x 4KB = 128MB
    Inode 数 = 8192

  1 亿文件需要的块组:
    100,000,000 / 8192 = 12,208 个块组

  每个 Block Group 的空间分配:
    ┌──────────────────────────────────────────┐
    │ Superblock + GDT       ~12 KB  (3 个块)  │
    │ Block Bitmap           4 KB   (1 个块)   │
    │ Inode Bitmap           4 KB   (1 个块)   │
    │ Inode Table          2048 KB  (512 个块) │
    │ ─────────────────────────────────────── │
    │ 元数据合计            ~2068 KB           │
    │ 可用数据空间         ~126 MB            │
    │ 元数据占比             ~1.6%             │
    └──────────────────────────────────────────┘

  总磁盘空间:
    12,208 x 128MB = 1.56TB（最小磁盘需求）

  Inode 表空间:
    12,208 x 512 块 x 4KB = 25.6GB

  目录项空间:
    1亿 x 16B = 1.6GB
```

### 3.6 不同文件规模的开销对比

| 文件数量 | Inode 表 | 目录项 | 元数据合计 | 最小磁盘 | 元数据占比 |
|---|---|---|---|---|---|
| 100 万 | 256 MB | 16 MB | ~272 MB | ~15.6 GB | 1.7% |
| 1000 万 | 2.56 GB | 160 MB | ~2.72 GB | ~156 GB | 1.7% |
| 1 亿 | 25.6 GB | 1.6 GB | ~27.2 GB | ~1.56 TB | 1.7% |
| 10 亿 | 256 GB | 16 GB | ~272 GB | ~15.6 TB | 1.7% |

```
结论: 无论多少文件，元数据占比始终约 1.7%
      开销主要来自 Inode 表（256B/文件）
      目录项开销相对较小（平均 16B/文件）
```

## 4. 查找速度分析

### 4.1 核心结论

```
查找速度不取决于文件总数，而取决于单目录中的文件数

  1 亿文件分散在 10 万个目录中（每目录 1000 个文件）
    → 查找很快，每次 ~1ms

  1 亿文件集中在 1 个目录中
    → 查找极慢，可能需要数秒
```

### 4.2 三种场景对比

#### 场景 1: 文件分散在多层目录（推荐）

```
目录结构: /data/dir_001/sub_001/file_001
          1000 个一级目录 x 100 个二级目录 x 1000 个文件
          = 1 亿文件，每目录 1000 个

查找 /data/dir_500/sub_030/file_888:
  → 4 层路径 = 4 次目录查找
  → 每个目录 1000 项，占 1-2 个数据块
  → HTree 索引: 1 次 B-Tree 查找
  → 总 I/O: 8 次（4 层 x 2，缓存命中时 0 次）
  → 延迟: ~1-2ms（缓存命中 ~100us）

结论: 查找速度与文件总数无关，只与目录深度和每目录文件数有关
```

#### 场景 2: 1 万文件在单个目录

```
目录结构: /data/file_00001 ~ file_10000

查找 /data/file_5555:
  → 1 层路径 = 1 次目录查找
  → 目录 1 万项，占 3-4 个数据块
  → HTree: 1 次哈希查找，读 2-3 个块
  → 总 I/O: 2-3 次
  → 延迟: ~0.5-1ms（缓存命中 ~50us）

结论: 单目录 1 万文件，查找仍然很快
```

#### 场景 3: 100 万文件在单个目录（不推荐）

```
目录结构: /data/file_000001 ~ file_1000000

查找 /data/file_555555:
  → 1 层路径 = 1 次目录查找
  → 目录 100 万项，占 ~300 个数据块
  → HTree: 1 次哈希查找，但树更深（3-4 层）
  → 总 I/O: 4-6 次
  → 延迟: ~2-5ms（缓存未命中 ~10-20ms）

列出目录内容（ls）:
  → 需要遍历所有 100 万个目录项
  → 读取 ~300 个数据块
  → 延迟: ~1-5 秒（取决于磁盘速度）

结论: 单目录超过 10 万文件后，ls 等操作明显变慢
      单个文件查找通过 HTree 仍可接受
      但整体体验下降
```

### 4.3 HTree 对查找的影响

```
目录文件数 vs 查找性能:

  文件数        查找 I/O 次数    延迟（磁盘）    说明
  ────────────────────────────────────────────────────
  < 100         1-2            ~0.1ms        单块目录，无需 HTree
  100-10000     2-3            ~0.5ms        HTree 2-3 层
  1万-10万      3-4            ~1-2ms        HTree 3-4 层
  10万-100万    4-6            ~2-5ms        HTree 更深，哈希冲突增多
  > 100万       5-10           ~5-20ms       哈希冲突严重，性能退化

HTree 查找过程:
  hash("filename") → 遍历 B-Tree → 找到叶子节点 → 线性搜索
  平均 O(1)，最坏 O(n)（所有文件哈希冲突时，极端罕见）
```

### 4.4 查找完整路径的 I/O 流程

```
查找 /a/b/c/file.txt（假设所有目录各 1000 个文件）:

  ┌──────────────────────────────────────────────────────┐
  │ Step 1: 读根目录 inode（inode 2，已知）               │
  │ Step 2: 在根目录中查找 "a" → 得到 inode 12345         │
  │ Step 3: 读 /a 的 inode（inode 12345）                │
  │ Step 4: 在 /a 中查找 "b" → 得到 inode 67890          │
  │ Step 5: 读 /a/b 的 inode（inode 67890）              │
  │ Step 6: 在 /a/b 中查找 "c" → 得到 inode 11111        │
  │ Step 7: 读 /a/b/c 的 inode（inode 11111）            │
  │ Step 8: 在 /a/b/c 中查找 "file.txt" → 得到 inode 99999│
  │ Step 9: 读 file.txt 的 inode → 得到 Extent 映射      │
  │ Step 10: 根据 Extent 读取数据块                       │
  └──────────────────────────────────────────────────────┘

  最坏情况（全部未缓存）: 10 次磁盘 I/O
  最好情况（全部缓存命中）: 0 次磁盘 I/O（~几十 us）
  典型情况（元数据缓存命中，数据未命中）: 1 次磁盘 I/O

  Linux VFS 的 dentry cache 和 inode cache 会缓存:
    - 最近访问的目录项（dentry cache）
    - 最近访问的 inode（inode cache）
    - 热点数据的页缓存（page cache）
  二次访问同一文件通常 0 次磁盘 I/O
```

```
│    → 每目录 100 个文件，查找极快                          │
│    → ls 每个目录瞬间完成                                 │
│    → 缺点: 目录层级多（100 万个子目录）                   │
│                                                          │
│  方案 2: 三级目录（均衡）                                  │
│    /data/{00-99}/{0000-9999}/{000000-999999}            │
│    100 个一级 x 1 万个二级 x 1 万个三级                   │
│    → 每目录 1 万文件，查找快                              │
│    → ls 可接受（1-2 秒）                                 │
│    → 目录层级适中                                        │
│                                                          │
│  方案 3: 按业务分目录                                     │
│    /data/{user_id}/{date}/{filename}                    │
│    → 按用户、日期等业务维度自然分片                       │
│    → 符合业务查询模式                                    │
│    → 每目录文件数可控                                    │
│                                                          │
├──────────────────────────────────────────────────────────┤
│  核心原则:                                                │
│    1. 每个目录不超过 1 万文件                             │
│    2. 目录深度不超过 4 层                                 │
│    3. 文件名长度适中（8-32 字节）                         │
│    4. 避免频繁创建/删除文件（产生碎片和目录块空洞）       │
│    5. 超过 10 亿文件考虑分布式文件系统                    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 6. 与分布式文件系统的元数据对比

```
┌──────────────────────────────────────────────────────────────┐
│                  元数据管理方式对比                             │
├──────────────┬──────────┬──────────┬──────────┬──────────────┤
│ 对比项        │ ext4     │ CephFS   │ 3FS      │ DAOS         │
├──────────────┼──────────┼──────────┼──────────┼──────────────┤
│ 元数据存储    │ 本地磁盘  │ MDS+RADOS│ FDB      │ SCM+VOS      │
│ 元数据扩展    │ 单机限制  │ MDS集群  │ 水平扩展  │ 水平扩展     │
│ 最大文件数    │ ~40亿/FS │ 几乎无限  │ 几乎无限  │ 几乎无限     │
│ 单机文件查找  │ HTree    │ MDS缓存  │ Meta缓存 │ Meta缓存     │
│ 1亿文件开销   │ ~27GB    │ 取决于MDS│ 取决于FDB│ 取决于SCM   │
│ 跨节点一致性  │ 无       │ Paxos    │ FDB SSI  │ 对象版本     │
└──────────────┴──────────┴──────────┴──────────┴──────────────┘

ext4 的元数据限制:
  - 单机 inode 数上限约 40 亿（32 位 inode 号）
  - 元数据查找依赖本地磁盘 I/O
  - 无法跨节点水平扩展
  - 适合: 单机/单节点场景，文件数 < 1 亿

分布式文件系统的优势:
  - 元数据服务可水平扩展
  - 文件数几乎不受限
  - 元数据缓存在内存中，查找更快
  - 适合: 集群环境，文件数 > 1 亿
```
