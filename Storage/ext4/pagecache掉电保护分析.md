# Page Cache 掉电丢失保护分析

---

## 1. Page Cache 掉电数据会丢失吗

**会丢失。** Page Cache 是内存，掉电即消失。`write()` 返回成功不代表数据已持久化。

### 1.1 数据持久化链路

```
write() 返回 ──▶ Page Cache (内存, dirty) ──▶ Writeback ──▶ 磁盘 Cache ──▶ NAND/盘片
                   ↑                           ↑               ↑              ↑
                   掉电必丢                    掉电必丢        可能丢失        真正安全
```

数据要真正持久化，需要经历四道关卡，任何一关掉电都会丢失：

```
┌──────────────────────────────────────────────────────────────────┐
│                  掉电时数据在哪, 丢不丢                            │
│                                                                   │
│  阶段 1: write() 刚返回                                          │
│  ─────────────────────────                                       │
│  数据只在 Page Cache (内存), 标记为 dirty                         │
│  → 掉电 → 数据丢失                                               │
│                                                                   │
│  阶段 2: Writeback 已发起                                         │
│  ─────────────────────────                                       │
│  dirty page 正在通过 Block Layer 写向磁盘                          │
│  → 数据在内核 DMA Buffer 中或正在传输                              │
│  → 掉电 → 数据丢失                                               │
│                                                                   │
│  阶段 3: 已到达磁盘控制器 (Disk Cache)                            │
│  ─────────────────────────                                       │
│  数据在 SSD/HDD 的 volatile DRAM Cache 中                        │
│  → 如果磁盘开启 Write Back Cache (默认)                           │
│  → 掉电 → 数据可能丢失                                           │
│                                                                   │
│  阶段 4: 已写入 NAND 介质 / 磁盘盘片                              │
│  ─────────────────────────                                       │
│  → 掉电 → 数据安全, 不丢失                                       │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 Page Cache 何时刷盘

```
┌──────────────────────────────────────────────────────────────┐
│  触发 writeback 的时机                                       │
│                                                              │
│  1. 定期回写 (周期 30s)                                     │
│     内核 flusher 线程定期扫描 dirty pages 并写回              │
│     → 最坏情况数据在内存中最多停留 30 秒                     │
│                                                              │
│  2. dirty_ratio 阈值 (默认内存的 10%)                        │
│     当 dirty page 数量超过阈值, 强制回写                      │
│     → write() 可能被阻塞, 等待 writeback 完成                │
│                                                              │
│  3. 应用主动调用 fsync() / fdatasync()                       │
│     → 强制将指定文件的所有 dirty pages 刷到磁盘              │
│     → 这是应用保证持久化的标准手段                             │
│                                                              │
│  4. sync() 系统调用                                          │
│     → 刷新所有文件的所有 dirty pages                          │
│     → 非常慢, 很少直接使用                                    │
│                                                              │
│  5. 内存紧张时 (kswapd)                                      │
│     系统需要回收内存页时, dirty pages 必须先写回再回收        │
│                                                              │
│  6. close() 文件关闭                                         │
│     不保证刷盘! close() 不等于 fsync()                        │
│     dirty page 仍然留在 Page Cache 中等待 writeback           │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 应用如何保证数据不丢

```
┌──────────────────────────────────────────────────────────────────┐
│                 保证持久化的手段 (从弱到强)                        │
│                                                                   │
│  1. 什么都不做 (依赖 writeback)                                  │
│     write() 后直接返回, 最快                                     │
│     掉电可能丢最多 30s 的数据                                    │
│     适用: 日志类 (可容忍少量丢失)                                 │
│                                                                   │
│  2. fsync(fd)                                                    │
│     将该文件的 dirty pages + 元数据全部刷到磁盘, 等待完成        │
│     适用: 数据库 (每个事务提交时)                                 │
│                                                                   │
│  3. fdatasync(fd)                                                │
│     与 fsync 类似, 但不强制刷 inode 元数据 (只刷数据块)          │
│     比 fsync 稍快                                                │
│                                                                   │
│  4. O_SYNC (open 时指定)                                        │
│     每次 write() 自动等数据落盘后才返回                          │
│     等效于 write() + fsync() 的原子操作                          │
│     性能极慢, 每次 write 都要等磁盘 IO                           │
│                                                                   │
│  5. O_DSYNC                                                      │
│     与 O_SYNC 类似, 但不保证元数据更新 (类似 fdatasync)         │
│                                                                   │
│  6. msync() + mmap()                                            │
│     mmap 写入后调用 msync(MS_SYNC) 强制刷盘                     │
│     适用: 内存映射文件的场景                                      │
└──────────────────────────────────────────────────────────────────┘
```

### 1.4 close() 不等于 fsync()

```
┌──────────────────────────────────────────────────────────────┐
│  常见误解: close() 会把数据刷到磁盘                           │
│                                                              │
│  实际行为:                                                   │
│  close() = 释放文件描述符 + 减少 inode 引用计数               │
│  如果引用计数不为 0 (还有其他 fd 打开), 什么都不做            │
│  即使引用计数为 0, dirty page 仍在 Page Cache 中              │
│  要等 flusher 线程定期回写或内存紧张时才刷盘                  │
│                                                              │
│  正确做法:                                                   │
│  fsync(fd) → close(fd)   保证数据持久化后再关闭              │
│  或者:                        close(fd) + 等 30s (不靠谱)    │
└──────────────────────────────────────────────────────────────┘
```

## 2. SSD 是否有 PLP

**不一定，取决于 SSD 等级。** 消费级 SSD 通常没有 PLP，企业级 SSD 基本都有。

### 2.1 各等级 SSD 的 PLP 配置

```
┌──────────────────────────────────────────────────────────────────┐
│                    SSD PLP 配置情况                               │
│                                                                   │
│  消费级 SSD (SATA / NVMe):                                       │
│  ────────────────────────────                                    │
│  例: Samsung 870 EVO, WD Blue SN580, Intel 670p                 │
│  PLP: 无 (极少数高端消费级有)                                    │
│  掉电: DRAM Cache 数据直接丢失                                   │
│  价格: 几百 ~ 一两千                                             │
│                                                                   │
│  企业级 SATA SSD:                                                │
│  ────────────────                                                │
│  例: Samsung SM883, Intel S4520, Micron 5400                    │
│  PLP: 有 (钽电容, 掉电维持数十毫秒)                              │
│  价格: 几千                                                      │
│                                                                   │
│  企业级 NVMe SSD:                                                │
│  ─────────────────                                               │
│  例: Samsung PM9A3, Intel P4610, Micron 7450, Kioxia CM7       │
│  PLP: 有 (电容/钽电容, 维持数十~数百毫秒)                       │
│  支持: in-flight write 完成 + FTL 元数据保护                     │
│  价格: 几千 ~ 上万                                                │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 PLP 工作原理

```
┌──────────────────────────────────────────────────────────────────┐
│                 有 PLP 的 SSD 掉电时发生了什么                    │
│                                                                   │
│  正常运行时:                                                      │
│  ─────────────                                                  │
│  ┌─────────┐    ┌───────────┐    ┌──────────────┐              │
│  │ DRAM    │    │ NAND      │    │ 电容/钽电容  │              │
│  │ Cache   │───▶│ (写入中)  │    │ (满电待命)   │              │
│  └─────────┘    └───────────┘    └──────┬───────┘              │
│                                        │                        │
│  掉电瞬间:                         电压检测器                   │
│  ─────────────                      检测到电压下降              │
│                                        │                        │
│  ┌─────────┐                    ┌──────▼──────┐              │
│  │ DRAM    │                    │ NAND Ctrl   │              │
│  │ (有数据)│                    │ 进入紧急模式 │              │
│  └────┬────┘                    └──────┬──────┘              │
│       │                              │                        │
│       └──▶ ┌─────────────────────────┐│                        │
│            │ 1. 完成当前 NAND 页写入 ││                        │
│            │ 2. 将 DRAM Cache 全部  ││                        │
│            │    刷入 NAND            ││                        │
│            │ 3. 更新 FTL 元数据     ││                        │
│            │ 4. 标记 NAND 块安全     ││                        │
│            └─────────────────────────┘│                        │
│                                       │                        │
│                              电容放电维持数十~数百毫秒            │
│                                       ▼                        │
│                                  安全关机, 数据完整              │
│                                                                   │
│  时间估算:                                                        │
│  - 大多数企业级 SSD 电容能维持 50~200ms                           │
│  - DRAM 容量通常 1~4GB, 写入带宽 1~3GB/s                         │
│  - 50ms 内可刷完数十 ~ 数百 MB, 远超 DRAM 容量                   │
│  - 足够完成所有 in-flight 操作                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 2.3 无 PLP 的 SSD 掉电风险

```
┌──────────────────────────────────────────────────────────────────┐
│              无 PLP 的消费级 SSD 掉电风险                          │
│                                                                   │
│  风险 1: DRAM Cache 数据丢失                                      │
│  ────────────────────────────                                    │
│  应用以为 fsync() 返回了 = 数据已持久化                          │
│  但实际还在 SSD 的 DRAM 中, 掉电就没了                           │
│  → 静默数据丢失, 应用无感知                                       │
│  严重程度: 高                                                     │
│                                                                   │
│  风险 2: NAND 页写入撕裂                                          │
│  ────────────────────────────                                    │
│  NAND 一个页 (16KB) 掉电时正在写 → 该页内容损坏                 │
│  SSD 固件有 CRC 校验 → 检测到后标记该页为坏块                   │
│  → 之前写入的合法数据丢失 (该页对应的内容)                        │
│  严重程度: 中                                                     │
│                                                                   │
│  风险 3: FTL 元数据损坏                                           │
│  ────────────────────────────                                    │
│  FTL 维护 LBA → PBA 映射表, 掉电时正在更新 → 映射表不一致       │
│  严重时 SSD 需要断电恢复 (Power-On Recovery)                     │
│  极端情况: SSD 变砖 (需 Secure Erase 或 RMA)                    │
│  严重程度: 高                                                     │
│                                                                   │
│  消费级 SSD 固件的防护措施:                                       │
│  ─────────────────────────────                                   │
│  1. 保守写入: 不同时写多个块, 减少撕裂范围                       │
│  2. 元数据冗余: FTL 映射表存储多份                               │
│  3. 掉电恢复扫描: 上电时检测不一致并修复                         │
│  4. 写入原子性标记: 只在页完整写入后标记为有效                    │
│                                                                   │
│  → 能降低风险但不能 100% 保证, 极端情况仍可能丢数据             │
└──────────────────────────────────────────────────────────────────┘
```

### 2.4 fsync() 在不同 SSD 上的实际安全边界

```
┌──────────────────────────────────────────────────────────────────┐
│              fsync() 返回后数据到底安全了吗                        │
│                                                                   │
│  有 PLP 的企业级 SSD:                                             │
│  ─────────────────────────                                       │
│  fsync() → OS 刷 Page Cache → SSD 收到写命令                     │
│         → SSD 写入 NAND → 返回成功                               │
│  掉电时: 电容供电, 完成所有未完成的 NAND 写入                     │
│  结论: fsync() 返回 = 数据真正安全                                │
│                                                                   │
│  无 PLP 的消费级 SSD:                                             │
│  ─────────────────────────                                       │
│  fsync() → OS 刷 Page Cache → SSD 收到写命令                     │
│         → SSD 写入 DRAM Cache → 返回成功                         │
│  掉电时: DRAM Cache 数据丢失!                                    │
│  结论: fsync() 返回 ≠ 数据真正安全                                │
│                                                                   │
│  无 PLP 但关闭 Write Cache:                                        │
│  ────────────────────────────                                    │
│  hdparm -W 0 /dev/sda (SATA)                                    │
│  nvme write-cache set /dev/nvme0 0 (NVMe)                       │
│  fsync() → SSD 必须 NAND 写完才返回                              │
│  结论: fsync() 返回 = 数据安全 (但写入延迟增大)                   │
└──────────────────────────────────────────────────────────────────┘
```

## 3. 生产环境应对方案

```
┌──────────────────────────────────────────────────────────────────┐
│               生产环境如何应对掉电数据丢失                        │
│                                                                   │
│  方案 1: 采购企业级 SSD (推荐, 最省心)                            │
│  ─────────────────────────────────────                           │
│  直接买带 PLP 的企业盘                                           │
│  fsync() 返回即代表数据安全                                       │
│  适合: 数据库, 金融, 核心业务                                     │
│                                                                   │
│  方案 2: 关闭磁盘 Write Cache                                      │
│  ─────────────────────────                                       │
│  hdparm -W 0 /dev/sda        (SATA)                              │
│  nvme write-cache set /dev/nvme0 0  (NVMe)                      │
│  每次 fsync() 等待 NAND 写入完成                                 │
│  代价: 写入延迟增大, 无法利用 DRAM Cache 加速                    │
│  适合: 有 UPS 的环境                                              │
│                                                                   │
│  方案 3: UPS + 自动关机脚本                                       │
│  ────────────────────────────                                    │
│  掉电时 UPS 供电 → 触发关机脚本 → sync → 安全关机               │
│  给系统足够时间把 Page Cache 刷到磁盘                             │
│  适合: 通用服务器                                                  │
│                                                                   │
│  方案 4: 应用层 WAL 日志                                          │
│  ────────────────────────                                        │
│  数据库 WAL + group commit + fsync                               │
│  即使掉电丢部分数据, 重启后从 WAL 重放恢复                       │
│  适合: 所有需要数据持久化的应用                                   │
│                                                                   │
│  方案 5: RAID 卡带 BBU (电池备份)                                │
│  ────────────────────────────                                    │
│  RAID 卡的 Write Cache 有电池保护                                 │
│  掉电时电池维持 RAID 卡运行, 完成所有写入                         │
│  适合: 传统阵列架构                                               │
└──────────────────────────────────────────────────────────────────┘
```

## 4. 总结

| 问题 | 结论 |
|------|------|
| Page Cache 掉电会丢数据吗 | **会。** `write()` 返回成功仅表示数据到了内存 Page Cache, 掉电即丢失。需 `fsync()` 才能触发刷盘, 且刷盘后还要看磁盘是否有 Write Cache |
| SSD 是否有 PLP | **看等级。** 消费级 SSD 通常无 PLP, `fsync()` 返回不代表数据已入 NAND; 企业级 SSD 有电容保护, `fsync()` 返回 = 数据真正安全 |
| 最简生产建议 | 企业级 SSD + UPS + 应用层 WAL, 三管齐下 |
