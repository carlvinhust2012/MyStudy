# Ext4 Block Size 与非对齐写入处理

---

## 1. Ext4 的 Block Size 一定是 4KB 吗

不是，block size 在格式化时可配置，支持 1KB、2KB、4KB、64KB 四种。

### 1.1 Block Size 选项

```
mkfs.ext4 时的 block size 选项:
  ──────────────────────────────────────────────────────
  mkfs.ext4 -b 1024 /dev/sda1    → 1KB per block
  mkfs.ext4 -b 2048 /dev/sda1    → 2KB per block
  mkfs.ext4 -b 4096 /dev/sda1    → 4KB per block (默认)
  mkfs.ext4 -b 65536 /dev/sda1   → 64KB (需 bigalloc 特性)

  限制: block size 不能超过内存页大小 (通常 4KB)
        除非启用 bigalloc 特性 (cluster size 可以大于 page size)
```

### 1.2 为什么 4KB 是默认也是最佳选择

```
┌──────────────────────────────────────────────────────────────┐
│                 4KB Block Size 的三层对齐                      │
│                                                               │
│  CPU 层: Linux 内存页 = 4KB                                   │
│  ─────────────────────────────                               │
│  Page Cache 读写以页为单位                                     │
│  block size = page size → 零拷贝, 无需拆分/合并               │
│  如果 block=1KB → 一个 4KB 页包含 4 个 block, 需额外管理     │
│                                                               │
│  文件系统层: Ext4 最小分配单元                                │
│  ─────────────────────────────                               │
│  Inode 的 i_block[0..14] = 60 字节                           │
│  一个 Extent Entry = 12 字节 → 根节点最多 4 个 entry           │
│  4 个 entry 覆盖范围:                                         │
│    block=1KB → 4 × 32768 × 1KB = 128MB                      │
│    block=4KB → 4 × 32768 × 4KB = 512MB                      │
│  block 越大, 单个 inode 直接索引范围越大                       │
│                                                               │
│  存储层: SSD/HDD 物理扇区                                     │
│  ─────────────────────────────                               │
│  NVMe SSD 最小读写单位 = 4KB                                   │
│  现代 HDD 支持 4K 物理扇区 (Advanced Format)                  │
│  block size = 4KB → 直接下发, 无底层 read-modify-write        │
│  如果 block=1KB, SSD 需先读出 4KB, 改其中 1KB, 写回 4KB      │
│                                                               │
│  空间浪费 vs 碎片化平衡:                                      │
│  ─────────────────────────────                               │
│  block size 越大 → 内部碎片越多 (小文件浪费空间)              │
│  block size 越小 → 元数据越多 (Block Bitmap 更大, 管理开销)   │
│  4KB 对大多数场景是最佳平衡点                                  │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 不同 Block Size 的对比

| | 1KB | 2KB | 4KB | 64KB (bigalloc) |
|---|---|---|---|---|
| 最小分配 | 1KB | 2KB | 4KB | 64KB |
| 1KB 文件实际占用 | 1KB | 2KB | 4KB | 64KB |
| 单 inode 直接索引 | 128MB | 256MB | 512MB | 8GB |
| Block Bitmap 大小 | 每 8KB 数据 1 Byte | 每 16KB 数据 1 Byte | 每 32KB 数据 1 Byte | 每 512KB 数据 1 Byte |
| 与 SSD 对齐 | 否 (需 RMW) | 否 (需 RMW) | 是 | 是 |
| 适用场景 | 大量小文件 | 通用折中 | 通用 (默认) | 大文件 (视频/数据库) |

### 1.4 Cluster 与 Block 的区别

```
正常模式 (无 bigalloc): block = cluster, 最小分配 = 1 block
bigalloc 模式: cluster = N × block, 最小分配 = 1 cluster

  例: block=4KB, cluster=64KB (16 个 block 组成 1 个 cluster)

  写入 1KB 文件:
  ┌──────────────────────────────────────────────────┐
  │  正常模式: 分配 1 个 block (4KB), 浪费 3KB      │
  │  bigalloc: 分配 1 个 cluster (64KB), 浪费 63KB  │
  │                                                  │
  │  bigalloc 的意义:                                 │
  │  - 极大减少 Block Bitmap 大小                     │
  │  - 减少元数据操作次数                             │
  │  - 适合视频流/对象存储等大文件场景                │
  │  - 不适合大量小文件场景                           │
  └──────────────────────────────────────────────────┘
```

## 2. 非对齐写入如何处理

应用可以任意偏移、任意长度写入，但底层磁盘只能按 block 整块操作。中间的差距由 Page Cache 的 **Read-Modify-Write (RMW)** 机制弥合。

### 2.1 一个典型例子

```
应用请求: write(fd, buf, 5KB) at offset=3KB
  ─────────────────────────────────────────────
  ┌───────┬──────────────────────────┬─────────┐
  │ Block0│ Block1                   │ Block2  │ ...
  │       │                          │         │
  │旧数据 │ ← 3KB空隙 →│← 5KB 数据→ │ 旧数据  │
  │       │    不修改     覆盖这些   │         │
  └───────┴──────────────────────────┴─────────┘
       ↑ 非对齐起始       跨 2 个 block     ↑ 非对齐结束

  磁盘操作:
  ─────────────────────────────────────────────
  Block0: 读出旧 4KB → 修改后 1KB → 写回完整 4KB (RMW)
  Block1: 新数据 4KB → 直接整块写入 (无需 RMW)
  Block2: 不涉及 (3+5=8KB, 止于 Block1 末尾)
```

### 2.2 Read-Modify-Write 详细流程

```
┌──────────────────────────────────────────────────────────────────┐
│              非对齐写入的完整处理流程                              │
│                                                                   │
│  write(fd, buf, 5KB) at offset=3KB                               │
│                                                                   │
│  Step 1: VFS 层分解为页粒度                                      │
│  ─────────────────────────────                                   │
│  generic_perform_write() 将请求拆分:                             │
│                                                                   │
│  Page0 [0KB~4KB]:  写入范围 [3KB, 4KB) → 后 1KB (部分页)       │
│  Page1 [4KB~8KB]:  写入范围 [0KB, 4KB)  → 全 4KB  (完整页)      │
│                                                                   │
│  Step 2: 对 Page0 执行 Read-Modify-Write                         │
│  ─────────────────────────────                                   │
│                                                                   │
│  case A: Page0 不在 Page Cache 中 (cache miss)                   │
│  ┌──────────────────────────────────────┐                        │
│  │  1. block_write_begin() 被调用       │                        │
│  │  2. 从磁盘读取整个 Page0 (4KB)       │                        │
│  │  3. 将 buf 的前 1KB 拷贝到 3072~4095 │                        │
│  │  4. Page0 标记为 dirty               │                        │
│  │                                      │                        │
│  │  Page0 内存布局:                     │                        │
│  │  [旧数据 3KB] [新数据 1KB]           │                        │
│  └──────────────────────────────────────┘                        │
│                                                                   │
│  case B: Page0 已在 Page Cache 中 (cache hit)                    │
│  ┌──────────────────────────────────────┐                        │
│  │  1. 无需读磁盘                       │                        │
│  │  2. 直接将 buf 拷贝覆盖 3072~4095    │                        │
│  │  3. Page0 标记为 dirty               │                        │
│  └──────────────────────────────────────┘                        │
│                                                                   │
│  Step 3: 对 Page1 执行完整写入 (无需 RMW)                        │
│  ─────────────────────────────                                   │
│  ┌──────────────────────────────────────┐                        │
│  │  1. 如果 Page1 不在 cache 中         │                        │
│  │     直接分配新 page, 填入全部 4KB    │                        │
│  │     不需要先读旧数据!                │                        │
│  │  2. 标记 dirty                       │                        │
│  │                                      │                        │
│  │  Page1 内存布局:                     │                        │
│  │  [新数据 4KB 全部]                   │                        │
│  └──────────────────────────────────────┘                        │
│                                                                   │
│  Step 4: Writeback 落盘                                           │
│  ─────────────────────────────                                   │
│  flusher 线程将 dirty pages 写回磁盘:                            │
│  Page0 → 完整 4KB 写入 Block0 (旧+新混合)                       │
│  Page1 → 完整 4KB 写入 Block1 (全为新数据)                       │
│  → 磁盘永远收到的是完整的 4KB 块                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.3 各层看到的写入请求

```
┌───────────────────────────────────────────────────────────────────┐
│              各层视角下的写入                                       │
│                                                                    │
│  应用层:      write(fd, buf, 5KB, offset=3KB)                     │
│               任意偏移, 任意长度, 无限制                            │
│                                                                    │
│       ▼ VFS generic_perform_write()                               │
│                                                                    │
│  VFS 层:      分解为页粒度操作                                     │
│               Page0: offset_in_page=3072, bytes=1024 (部分页)     │
│               Page1: offset_in_page=0,    bytes=4096 (完整页)     │
│               对部分页调用 block_write_begin() → 触发 RMW          │
│                                                                    │
│       ▼ Page Cache                                                 │
│                                                                    │
│  Page Cache:   每页固定 4KB, 始终按完整页操作                      │
│               部分写入 → 读旧数据, 修改部分, 标记 dirty             │
│               完整写入 → 直接填入, 标记 dirty (无需读旧数据)       │
│                                                                    │
│       ▼ ext4 ext4_writepages()                                    │
│                                                                    │
│  ext4:        以完整 block 为单位提交                               │
│               块分配 4KB 粒度, 永远整块分配                        │
│               writeback 时提交完整 4KB page                        │
│               块分配请求中也只有整块, 无碎片请求                    │
│                                                                    │
│       ▼ Block Layer blk-mq                                         │
│                                                                    │
│  Block Layer:  struct bio, 每个 segment = 4KB                     │
│                                                                    │
│       ▼ NVMe Driver                                                │
│                                                                    │
│  NVMe:        SQE: NLB=1, LBA=对应块号                            │
│               PRP 指向完整 4KB 物理页                              │
│               磁盘永远收到对齐的完整块                              │
│                                                                    │
│  结论: "非对齐"只在 Page Cache 层通过 RMW 处理,                   │
│        到达 ext4/Block Layer/NVMe 时已经完全对齐                  │
└───────────────────────────────────────────────────────────────────┘
```

### 2.4 O_DIRECT 模式下的非对齐写入

```
┌───────────────────────────────────────────────────────────────────┐
│                 O_DIRECT 绕过 Page Cache                           │
│                                                                    │
│  O_DIRECT 特点: 直接 I/O, 绕过 Page Cache                        │
│                                                                    │
│  write(fd, buf, 5KB, offset=3KB, O_DIRECT)                       │
│                                                                    │
│  结果: 返回 -EINVAL, 直接被内核拒绝                                │
│                                                                    │
│  Linux 对 O_DIRECT 的硬性限制:                                     │
│  ┌─────────────────────────────────────────────────────┐          │
│  │  1. buf 地址必须页对齐 (4KB 边界)                   │          │
│  │  2. offset 必须是 512B 的整数倍 (扇区对齐)          │          │
│  │  3. 长度必须是 512B 的整数倍                        │          │
│  └─────────────────────────────────────────────────────┘          │
│                                                                    │
│  原因:                                                            │
│  - 没有 Page Cache 做 RMW, 磁盘无法处理非对齐                     │
│  - 必须由应用自己保证对齐                                         │
│                                                                    │
│  数据库的做法:                                                    │
│  ┌─────────────────────────────────────────────────────┐          │
│  │  MySQL InnoDB: page size = 16KB, 所有读写 16KB 对齐 │          │
│  │  PostgreSQL:  page size = 8KB,  所有读写 8KB 对齐   │          │
│  │  RocksDB:      block size 可配, 默认 4KB/8KB 对齐    │          │
│  │                                                     │          │
│  │  数据库自己维护 buffer pool, 确保页对齐              │          │
│  │  通过自己的 WAL 日志保证崩溃一致性                    │          │
│  │  不依赖文件系统的 Page Cache 和 ordered journal      │          │
│  └─────────────────────────────────────────────────────┘          │
└───────────────────────────────────────────────────────────────────┘
```

### 2.5 非对齐写入的崩溃一致性风险

```
┌───────────────────────────────────────────────────────────────────┐
│              非对齐写入 (RMW) 的崩溃风险                            │
│                                                                    │
│  场景: Page0 = [旧数据 A (3KB) | 旧数据 B (1KB)]                 │
│         要修改为 [旧数据 A (3KB) | 新数据 C (1KB)]                │
│                                                                    │
│  崩溃时间点            磁盘状态            一致性                   │
│  ─────────────────────────────────────────────────                │
│  RMW 之前              [A | B] 原始状态     一致 (无修改)         │
│  写入一半 (撕裂写)     [A | 半C半B]        不一致!                │
│  完整写入后            [A | C] 目标状态     一致                   │
│                                                                    │
│  撕裂写产生的 "半新半旧" 数据:                                    │
│  - 如果 C 是数据库页, 会导致数据库校验失败                        │
│  - 如果 C 是普通文件, 用户会看到损坏的内容                        │
│                                                                    │
│  Ext4 的保护:                                                    │
│  ────────────────────                                           │
│  data=ordered (默认):                                             │
│    先写数据块到最终位置 → 再 commit 元数据日志                    │
│    RMW 后的完整页在日志 commit 之前写回磁盘                       │
│    如果数据写入后日志 commit 前崩溃:                              │
│      → 重启后可能看到 [A | C] (数据已落盘但无对应元数据)         │
│      → 实际上旧元数据指向 [A | B], 所以文件看到的是旧数据         │
│      → 不会出现半新半旧                                          │
│    如果 RMW 撕裂写入 (半页):                                      │
│      → 元数据未更新 → 文件仍然看到旧数据 → 一致                   │
│      → 但物理磁盘上可能有垃圾数据, 不过文件系统逻辑上安全          │
│                                                                    │
│  data=journal:                                                    │
│    数据先进 journal → 再落盘 → 最安全                             │
│    journal 写入是顺序的, 不太会撕裂                               │
│    但 64MB 数据过 journal 性能很差                                │
│                                                                    │
│  底层硬件保护:                                                    │
│  ────────────────────                                           │
│  NVMe SSD:                                                        │
│    单个 4KB 写入在 SSD 内部有原子性保证                           │
│    (内部有电容/PLP 保护, 不会写半块)                              │
│    → RMW 写回的完整 4KB 要么成功要么失败, 不会撕裂               │
│                                                                    │
│  HDD:                                                             │
│    4KB = 8 个 512B 扇区                                          │
│    掉电时可能部分扇区写入成功, 部分失败                           │
│    → 存在撕裂写风险                                               │
│    → 依赖文件系统层 (JBD2) 的保护                                │
└───────────────────────────────────────────────────────────────────┘
```

## 3. 总结

| 问题 | 结论 |
|------|------|
| Ext4 一定是 4KB block 吗 | 不是, 支持 1K/2K/4K/64K, 但 4KB 是默认值也是最佳选择, 因为与 CPU 页/SSD 扇区天然对齐 |
| 非对齐写入如何处理 | Page Cache 层通过 Read-Modify-Write 将非对齐转换为完整页写入; 到达 ext4 和磁盘时已完全对齐; O_DIRECT 下直接拒绝非对齐请求, 要求应用自行保证 |
