# 阿里云文件存储、块存储与对象存储的架构关系解析

阿里云的块存储（EBS）、文件存储（NAS）、对象存储（OSS）并非独立的存储系统，而是基于同一个自研底层分布式存储底座——盘古（Pangu）构建的三种不同形态的存储服务，整体呈现“一底座、三形态、多协议”的分层架构。三者共享底层存储资源与核心能力，仅在上层接口、协议适配和功能封装上存在差异，以满足不同业务场景的需求。本文将从底层到上层，详细解析三者的架构关联、核心差异及协同逻辑。

# 一、底层统一底座：盘古分布式存储（Pangu）

盘古（Pangu）是阿里云所有云存储服务的统一持久化层，承担着数据存储、可靠性保障、水平扩展等核心职责，是三种存储服务能够实现高效协同的基础。

## 1\.1 盘古的核心定位与能力

- **统一存储池**：盘古通过大量ChunkServer（数据节点）和MetaServer（元数据集群），构建全局分布式存储池，所有上层存储服务（EBS/NAS/OSS）的数据最终都落到该存储池中，实现资源的统一调度与复用。

- **高性能存储引擎**：盘古2\.0及以上版本采用“裸NVMe SSD \+ 自研用户态轻量存储栈（USSOS）”架构，直接管理裸盘的物理地址（LBA），绕过内核文件系统（如ext4/XFS），避免内核拷贝、中断等性能损耗，最大化发挥SSD的IO性能。

- **核心基础能力**：原生支持多副本冗余、EC纠删码、快照、强一致性、水平扩容等能力，为上层服务提供统一的可靠性与可扩展性保障；提供Append\-Only（顺序写）和Random Access（随机读写）两种基础接口，适配不同上层服务的IO需求。

## 1\.2 盘古的底层部署架构

盘古集群主要由两类节点构成：

- **ChunkServer**：负责数据的实际存储与IO处理，直接管理裸NVMe盘，通过USSOS实现地址映射、空间管理、坏块处理、校验等功能，是数据面的核心节点。

- **MetaServer**：负责元数据的管理与调度，包括数据分片位置、权限信息、快照记录等，保障元数据的高可用与高效访问，是控制面的核心节点。

# 二、上层三大存储服务：接口与功能的分化

基于盘古的统一底座，阿里云通过不同的上层封装，形成了块存储（EBS）、文件存储（NAS）、对象存储（OSS）三种服务形态，分别适配块级、文件级、对象级的访问需求，三者共享底层存储资源，但接口、协议和功能各有侧重。

## 2\.1 块存储 EBS（云盘）

### 2\.1\.1 核心定位与接口

EBS（Elastic Block Storage）是面向云服务器（ECS）的块级存储服务，对外暴露标准块设备接口（如/dev/vd），可直接挂载到ECS实例，支持格式化、分区、随机读写，相当于“云服务器的本地硬盘”，主要用于承载低延迟、高IOPS的随机读写场景。

### 2\.1\.2 与盘古的架构关联

EBS是盘古上层最简洁的封装之一，核心是将盘古的存储资源映射为虚拟磁盘（VD）：

- 在盘古存储池中创建一个大文件/对象，将其映射为一块虚拟磁盘（VD），每个EBS云盘对应一个独立的VD。

- EBS层不负责底层数据存储，仅专注于块级功能封装，包括块级快照、克隆、加密、QoS（IO速率限制）、IO调度等，确保满足系统盘、数据库盘等场景的需求。

- IO路径：ECS实例 → EBS用户态/内核态驱动 → 盘古ChunkServer → 裸NVMe盘（无内核文件系统，通过USSOS直接访问）。

### 2\.1\.3 典型应用场景

ECS系统盘、数据库（MySQL、PostgreSQL等）数据盘、需要低延迟随机读写的业务场景（如OLTP数据库、虚拟化环境）。

## 2\.2 文件存储 NAS

### 2\.2\.1 核心定位与接口

NAS（Network Attached Storage）是面向多客户端共享的文件级存储服务，对外暴露标准的NFS（Linux/Unix）、SMB（Windows）文件协议，支持目录结构、文件权限管理、多客户端并发挂载，相当于“云环境中的共享文件夹”，主要用于解决多节点共享存储的需求。

### 2\.2\.2 与盘古的架构关联

NAS在盘古的基础上，额外构建了文件系统层和协议网关，实现文件级的访问与管理：

- 基于盘古的存储资源，构建分布式文件系统，提供全局命名空间、目录树、文件锁、权限控制等文件级核心能力。

- NAS层包含协议网关（NFS/SMB网关），负责将客户端的文件请求（如读写文件、创建目录）转换为盘古能够识别的接口请求，同时处理缓存加速、协议适配等逻辑。

- IO路径：客户端（ECS、本地服务器等） → NAS协议网关 → 盘古MetaServer（元数据查询） → 盘古ChunkServer → 裸NVMe盘。

### 2\.2\.3 典型应用场景

多ECS实例共享存储（如Web服务器集群、应用服务器集群）、媒体编辑、高性能计算（HPC）、大数据分析、容器集群共享存储等。

## 2\.3 对象存储 OSS

### 2\.3\.1 核心定位与接口

OSS（Object Storage Service）是面向海量非结构化数据的对象级存储服务，对外暴露RESTful HTTP API（兼容S3协议），以“桶（Bucket）\+ 对象（Object）”的形式组织数据，支持元数据管理、生命周期管理等功能，主要用于承载海量、高吞吐、一次写入多次读取（WORM）的非结构化数据。

### 2\.3\.2 与盘古的架构关联

OSS与盘古的适配性最高，盘古原生支持对象存储模型，OSS仅需在上层封装HTTP协议与业务功能：

- 盘古底层天然支持对象的分片存储、多副本冗余，OSS直接复用这些能力，将用户上传的对象（图片、音视频、文档等）分片后存储到盘古存储池中。

- OSS层负责HTTP协议适配、签名鉴权、多版本管理、跨域访问、CDN加速、归档分层（标准、低频、归档）等业务功能，满足海量数据的存储与访问需求。

- IO路径：客户端（浏览器、APP、ECS等） → OSS网关（HTTP协议解析、鉴权） → 盘古ChunkServer → 裸NVMe盘。

### 2\.3\.3 典型应用场景

图片/音视频存储、静态网站托管、数据备份与归档、AI训练数据湖、日志存储、云原生应用的静态资源存储等。

# 三、三大存储服务的架构关系总览

## 3\.1 分层架构图（从上到下）

```Plain Text
用户/应用层
    ↓       ↓       ↓
┌─────────┬─────────┬─────────┐
│  EBS    │  NAS    │  OSS    │  上层服务：不同接口/协议封装
└─────────┴─────────┴─────────┘
            ↓
┌─────────────────────────────┐
│     盘古（Pangu）           │  底层底座：统一分布式存储
└─────────────────────────────┘
            ↓
┌─────────────────────────────┐
│ 裸NVMe盘 + 用户态USSOS      │  物理层：直接管理裸盘，高性能IO
└─────────────────────────────┘
```

## 3\.2 核心联系与区别

### 3\.2\.1 核心联系

- 共享底层底座：三者均基于盘古分布式存储，数据最终都存储在盘古的全局存储池中，共享盘古的可靠性（多副本/EC）、扩展性、高性能等核心能力。

- 资源协同：底层存储资源可统一调度，例如EBS云盘的数据可通过快照备份到OSS，NAS文件可直接同步到OSS归档，实现数据的跨服务流转。

- 统一运维：底层盘古集群的运维（扩容、故障恢复、备份）统一进行，上层三种服务无需单独维护底层存储，降低运维成本。

### 3\.2\.2 核心区别

|对比维度|块存储 EBS|文件存储 NAS|对象存储 OSS|
|---|---|---|---|
|接口形态|标准块设备接口（/dev/vd）|NFS/SMB 文件协议|RESTful HTTP API（S3兼容）|
|访问模式|随机读写，低延迟，单客户端独占挂载|文件级共享，多客户端并发访问，支持目录树|对象级访问，高吞吐，一次写入多次读取|
|上层封装|块级快照、克隆、QoS、加密|文件系统、协议网关、缓存、权限管理|HTTP协议、鉴权、多版本、生命周期、CDN|
|典型场景|ECS系统盘、数据库盘、低延迟随机IO|多节点共享、HPC、媒体编辑|海量非结构化数据、备份归档、AI数据湖|

# 四、总结

阿里云的块存储（EBS）、文件存储（NAS）、对象存储（OSS），本质是“同一底层、三种封装”的存储服务。盘古作为统一的分布式存储底座，提供了底层的数据存储、可靠性和性能保障；而上层服务则通过不同的接口、协议和功能封装，分别满足块级、文件级、对象级的访问需求，覆盖从低延迟随机IO到海量高吞吐存储的全场景。

这种架构的优势在于：统一底层资源，降低存储成本和运维复杂度；上层服务独立封装，可灵活适配不同业务场景；同时支持跨服务数据流转，构建完整的存储生态，满足企业多样化的存储需求。
