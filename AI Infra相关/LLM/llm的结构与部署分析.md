
#### Token Embedding 详解

**本质不是"组合"，而是每个 token 一个专属向量。** Embedding 是一张二维查找表：

```
              维度 0  维度 1  维度 2  ...  维度 4095
            ┌───────┬───────┬───────┬─────┬────────┐
token_0 (的)  │  0.12 │ -0.34 │  0.56 │ ... │  0.78  │
token_1 (一)  │  0.23 │  0.45 │ -0.12 │ ... │ -0.33  │
token_2 (是)  │ -0.67 │  0.89 │  0.11 │ ... │  0.44  │
   ...        │  ...  │  ...  │  ...  │ ... │  ...   │
token_31999  │  0.01 │ -0.55 │  0.32 │ ... │ -0.91  │
            └───────┴───────┴───────┴─────┴────────┘
              ◄────────── 每行就是一个 token 的向量 ──────────►
              ◄──── 共 32000 行（词表大小）────►
              ◄──── 共 4096 列（隐藏维度）─────────────►
```

**工作方式：** 纯查表操作，没有计算过程。输入 token ID = 892，直接去第 892 行取出那个 4096 维向量。

**类比理解：** 类似 RGB 颜色模型用 3 个数字（R, G, B）表示颜色，Embedding 用 4096 个数字表示 token 的"语义坐标"，维度越高能表达的语义越丰富：

```
3维 (RGB):      能区分几百万种颜色
768维 (BERT):    能编码基本语义关系
4096维 (LLaMA):  能编码复杂的语法、语义、世界知识
```

#### Self-Attention 详解

Self-Attention（自注意力）是 Transformer 的核心机制——让每个 token 都能"回头看"其他 token，理解上下文关系。

**直观理解：** 句子"猫坐在垫子上"中，"坐"需要关注"猫"和"垫子"才能理解自己的角色。Self-Attention 就是为每个词自动找出"应该关注谁，关注多少"。

**计算过程（四步）：**

```
第一步: 生成 Q, K, V (通过三个线性变换)
  每个 token 产生三个向量:
  · Q (Query)  = "我在找什么?"    ← 查询
  · K (Key)    = "我能提供什么?"   ← 键
  · V (Value)  = "我的实际内容"    ← 值

第二步: 计算注意力分数 (Q 和 K 的点积)
  每个词的 Q 去和所有词的 K 做点积 → 分数越高越相关

         token_1("猫")  token_2("坐")  token_3("在")  token_4("垫子")
  Q₁·K      0.9           0.3           0.1          0.2
  Q₂·K      0.4           0.2           0.5          0.7

第三步: Softmax → 注意力权重 (转为概率分布)
         token_1("猫")  token_2("坐")  token_3("在")  token_4("垫子")
  权重₁    0.60           0.18          0.08          0.14   ← "猫"主要关注自己
  权重₂    0.12           0.10          0.28          0.50   ← "坐"主要关注"垫子"

第四步: 加权求和 V → 输出
  输出₁ = 0.60×V₁ + 0.18×V₂ + 0.08×V₃ + 0.14×V₄
  每个词的输出 = 所有词 Value 的加权组合，权重由相关性决定

公式: Attention(Q, K, V) = Softmax(Q × Kᵀ / √d) × V
                                 ↑
                           缩放因子，防止分数过大导致 Softmax 饱和
```

**Multi-Head Attention（多头注意力）：**

```
单头只能捕获一种关系。多头 = 多个视角并行观察:

  头1 (语法关系):  "苹果" ↔ "涨价"    (主语↔谓语)
  头2 (语义关系):  "苹果" ↔ "今天"    (实体↔时间)
  头3 (位置关系):  "涨价" ↔ "了"      (动作↔结果)
  ...

每个头独立计算 Attention，最后拼接合并:

  头1输出 ─┐
  头2输出 ─┤→ 拼接 → Linear → 最终输出
  头N输出 ─┘

典型配置: LLaMA-7B 有 32 个头，每头 128 维
```

**Causal Mask（因果掩码）：**

```
Decoder-only 模型中，每个 token 只能看到自己及之前的 token:

         token_1  token_2  token_3  token_4  token_5
token_1   0        ✕        ✕        ✕        ✕
token_2   0        0        ✕        ✕        ✕
token_3   0        0        0        ✕        ✕
token_4   0        0        0        0        ✕
token_5   0        0        0        0        0

推理时逐个生成 token，生成 token_3 时还没看到 token_4/5，
训练时也要保持一致，否则会"作弊"
```

**类比：** Self-Attention 就像一场会议讨论——Q 是每个人提出的问题，K 是每个人的专业标签，V 是实际观点。注意力分数 = 找到最对口的人，听更多他的意见。多头 = 同时从语法、语义、逻辑等多个角度讨论。因果掩码 = 每个人只能听到前面已发言的人。

#### Residual Add（残差连接）详解

Residual Add 就是把层的输入直接"短路"加到输出上——让信息可以跳过某些层直接传递。

**直观理解：**

```
没有残差连接:
  输入 → [Attention] → [FFN] → 输出
  信息必须经过每一层，层数越多，原始信息越容易"失真"

有残差连接:
  输入 ─────────────────────────┐
    │                           │
    └→ [Attention] → [FFN] ─────┤ (+) → 输出
                                │
  输出 = 原始输入 + 变换后的结果

  信息有两条路: 一条走变换，一条直接穿过
  即使变换层没学好，原始信息也不会丢失
```

**为什么要这样做？解决深层网络的梯度消失问题：**

```
  假设有 32 层，反向传播时梯度要连乘 32 次，越来越小 → 梯度消失

  残差连接: output = x + F(x)
  梯度:     ∂output/∂x = 1 + ∂F/∂x
                        ↑
                  恒有这个 "1"，梯度不会消失！

  类比: 普通网络 = 传话游戏，32人依次传话，最后变形严重
       残差网络 = 每人拿到原文+上一个人版本，可对照修正，不会越传越偏
```

**在 Transformer 中的位置：** 每层有两个残差连接：

```
  X ──────────────────┐
  │                    │
  └→ RMSNorm → Attention ──┤ (+) ──────────────┐
                          │                    │
                          └→ RMSNorm → FFN ────┤ (+) → 下一层

  第一个残差: 保留原始 token 上下文信息
  第二个残差: 在 Attention 基础上保留 FFN 前的信息
```

| 对比项 | 没有残差连接 | 有残差连接 |
|-------|------------|-----------|
| 可训练深度 | 10层左右就困难 | 100+ 层也稳定 |
| 梯度传播 | 容易消失 | 畅通无阻 |
| 层数增加效果 | 反而下降 | 持续提升 |

#### RMSNorm 详解

RMSNorm 是 LayerNorm 的简化版——去掉了均值计算，只做"幅度归一化"，更快更简单。

**为什么需要归一化？** Transformer 每层输出数值范围会不断变化，不经归一化会导致数值爆炸，训练不稳定。归一化的作用就是把数值拉回合理范围。

**LayerNorm（传统方式）：**

```
LayerNorm(x) = (x - mean) / std × γ + β

步骤1: 算均值    mean = (x₁ + x₂ + ... + xₙ) / n
步骤2: 算方差    var = ((x₁-mean)² + (x₂-mean)² + ...) / n
步骤3: 标准化    x_norm = (x - mean) / √var
步骤4: 缩放平移  x_out = γ × x_norm + β

特点: 既有均值归零，又有方差归一化，还有可学习参数 γ 和 β
缺点: 计算量大
```

**RMSNorm（简化版）：**

```
RMSNorm(x) = x / RMS(x) × γ

步骤1: 算均方根  RMS = √(mean(x²))
步骤2: 归一化  x_norm = x / RMS
步骤3: 缩放    x_out = γ × x_norm

去掉了: 均值计算(不减均值)、偏移参数 β(只保留缩放 γ)
类比: LayerNorm = 先平移到中心 + 拉伸到标准宽度 + 微调
     RMSNorm   = 不管中心在哪，只调整幅度 + 微调
```

| 对比项 | LayerNorm | RMSNorm |
|-------|-----------|---------|
| 步骤 | 4 步 | 2 步 |
| 速度 | 基准 | 快 7-10% |
| 可学习参数 | γ + β | 仅 γ |
| 效果 | — | 几乎相同 |
| 采用模型 | BERT, GPT-2 | LLaMA, Qwen, Mistral, DeepSeek |

**为什么够用？** 去掉均值归零对模型影响很小，幅度归一化才是稳定训练的关键。LLaMA 论文证实 RMSNorm 与 LayerNorm 效果相当但更快。

### 1.6 参数分布（以 Decoder-only 为例）

```
模型总参数 100%
│
├── Embedding 层         ~2-5%   词表大小 × 隐藏维度
│
├── Transformer 层       ~90-95%  (每层结构相同，乘以层数)
│   ├── Self-Attention   ~35%     Q, K, V, O 四个投影矩阵
│   └── FFN              ~65%     三个线性层 (SwiGLU)
│
└── LM Head              ~2-5%   隐藏维度 × 词表大小
                         (部分模型与 Embedding 共享权重)
```

#### Continuous Batching 详解

Continuous Batching 是推理框架中提升吞吐量的关键调度策略——**不要等一批请求全部完成再处理下一批，而是随时把新请求"插队"加入正在运行的批次中。**

**传统 Static Batching 的问题：**

```
凑齐一批，跑完一批，再凑下一批

请求A ████████████████░░░░░░░░░░░░░░░░░░  (短，早结束但被迫等待)
请求B ████████████████████████████████████  (长)
请求C ████████████░░░░░░░░░░░░░░░░░░░░░░  (短，早结束但被迫等待)
请求D ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  (等整批结束才加入)
       │← 整批一起结束 →│← 浪费！→│

问题: 短请求完成后 GPU 空等；新请求要等整批跑完，GPU 利用率低
```

**Continuous Batching 的做法：**

```
谁结束谁让位，新请求立即补上

请求A ████████████████
请求B ████████████████████████████████████
请求C ████████████
请求D             ██████████████            ← 立即补上
请求E                         ████████████ ← 又补上

GPU 始终在干活，没有空闲等待
```

| 场景 | Static Batching | Continuous Batching |
|------|----------------|---------------------|
| 餐厅 | 凑齐10个客人才一起上菜，吃得快的也要等吃得慢的结账 | 来一个就上菜，吃完就走，新客随时入座 |
| 洗衣房 | 等8台全洗完才放下一批 | 哪台洗完哪台立刻放新衣服 |

#### PagedAttention 详解

PagedAttention 是 vLLM 提出的 KV Cache 内存管理技术，灵感来自操作系统的**虚拟内存分页**——按需分配，不浪费。

**传统方式的问题——预分配导致大量浪费：**

```
每个请求预先分配最大长度的连续内存

请求A (实际生成了 10 个token，但预分配了 2048 的空间)
┌──┬──┬──┬──┬──┬──┬──┬──┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┐
│K1│V1│K2│V2│K3│V3│K4│V4│ │ │ │ │ │ │ │ │ │ │ │ │ │
└──┴──┴──┴──┴──┴──┴──┴──┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┘
 ████████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░
  实际使用(20%)          预留但浪费(80%)

问题: 必须按最大长度预分配连续内存 → 大量显存被浪费 → 并发量减少
```

**PagedAttention 的做法——分页按需分配：**

```
1. 将 KV Cache 切分为固定大小的 "页" (Page)
2. 不预分配，按需分配
3. 不要求连续内存，页可以分散存放
4. 用页表 (Page Table) 记录映射关系

请求A 的 KV Cache (生成了 10 个token):
  页表: [Page #3] → [Page #7] → null
  ┌────────┐       ┌────────┐
  │ Page#3 │  ...  │ Page#7 │   ← 只分配2页，不需要连续
  └────────┘       └────────┘

空闲页可随时分配给新请求:
  ┌────────┐ ┌────────┐ ┌────────┐
  │ Page#4 │ │ Page#6 │ │ Page#8 │   ← 回收的空闲页
  └────────┘ └────────┘ └────────┘
```

类比：传统方式 = "团队最多可能来20人，预留20间连续房间，实际只来了5人，15间空着无法给其他客人用"；PagedAttention = "来几个定几个，房间不需要挨着，退房立刻回收"。

| 对比项 | 传统方式 | PagedAttention |
|-------|---------|---------------|
| 显存利用率 | ~20-50% | ~95%+ |
| 同显存并发量 | 少 | 提升 2-4 倍 |
| 内存分配 | 连续分配 (易碎片化) | 分散分配 (无碎片) |

#### FlashAttention 详解

FlashAttention 是注意力计算的加速算法——通过优化 GPU 显存读写（IO）来加速 Self-Attention，**不改变计算结果**。

**传统 Attention 的问题——IO 成为瓶颈：**

```
Self-Attention: Softmax(Q × K^T) × V

步骤1: Q × K^T → S (N×N 矩阵)    → 写入显存
步骤2: Softmax(S) → P (N×N 矩阵)   → 读 S，写 P 到显存
步骤3: P × V → O                   → 读 P，写 O 到显存

问题: 中间矩阵 S、P 都是 N×N (N=序列长度)
      N=4096 时 N×N=16M 元素, 大量占用显存带宽
      GPU 计算很快，瓶颈在显存读写 (IO-bound)
```

**FlashAttention 的做法——利用 GPU 片上高速缓存：**

```
传统: 显存 ←→ 显存           (慢，HBM 带宽是瓶颈)
Flash: 显存 → SRAM → SRAM → 显存 (快，SRAM 比 HBM 快 20 倍+)

将 Q, K, V 分成小块 (Block):
  ┌──────────────────────────────┐
  │  GPU 芯片                    │
  │  ┌────────────────────┐      │
  │  │ SRAM (高速缓存)     │      │
  │  │  加载 Q_block      │      │
  │  │  加载 K_block      │      │
  │  │  计算 QK^T (在SRAM) │     │
  │  │  乘 V_block (在SRAM) │     │
  │  │  累加到输出         │      │
  │  └────────┬───────────┘      │
  │           │ (少量读写)         │
  └───────────┼──────────────────┘
              ↕
  ┌────────────────────┐
  │ HBM 显存 (大但慢)   │  只读写最终结果，不存中间 N×N 矩阵
  └────────────────────┘
```

| 对比项 | 传统 Attention | FlashAttention |
|-------|---------------|----------------|
| 显存占用 | O(N²) | O(N) |
| 计算速度 | 基准 | 快 2-4 倍 |
| 计算结果 | — | 完全相同 |
| 长序列支持 | N>2048 就吃力 | N=128K 也能跑 |

#### FlashAttention vs FlashInfer

两者都用于加速注意力计算，但定位不同：

| | FlashAttention | FlashInfer |
|--|--------------|------------|
| 类比 | 高速发动机的核心设计图纸 | 针对赛车比赛改装的成品引擎 |
| 层面 | 算法层 | 系统层 |
| 作者 | Tri Dao (Stanford, 2022) | FlashInfer 团队 (2024) |
| 适用 | 训练 & 推理通用 | 主要面向推理 |
| 覆盖 | 单一注意力算子 | 完整推理注意力工作流 |

**FlashAttention（基础算法）：** 通用注意力加速算法，通过 IO-aware tiling 减少 HBM 读写，是注意力加速的奠基之作，版本从 v1 迭代到 v3。

**FlashInfer（推理深度优化）：** 专为 LLM 推理服务设计，在 FlashAttention 基础上针对推理场景做了更多工程优化：
- 原生支持 Prefill + Decode 混合批次（同一批里有的请求在 Prefill，有的在 Decode）
- 原生支持 Paged KV Cache（与 vLLM 分页机制完美配合）
- 支持多种注意力变体（GQA, MQA, Sliding Window Attention 等）

```
现在 vLLM 等框架默认使用 FlashInfer 作为注意力后端，
因为它对推理场景做了更多针对性优化

┌─────────────────────────────────────────────┐
│             推理框架 (vLLM 等)                │
│   ┌──────────────────────────────────┐       │
│   │   FlashAttention (通用基础)       │       │
│   │          ↓  推理场景优化           │       │
│   │   FlashInfer (推理深度优化)  ←── 主流  │
│   └──────────────────────────────────┘       │
└─────────────────────────────────────────────┘
```

```
  │         │          │           │              │
```

### 3.2 Decode（解码）阶段

逐个生成 token，每个 token 依赖之前所有 token 的 KV Cache：

```
用户     客户端      API服务    推理引擎(GPU)     模型
  │         │          │           │              │
  │         │          │           │ (从Prefill    │
  │         │          │           │  获得KV Cache │
  │         │          │           │  + 第一个token)│
  │         │          │           │              │
  │         │          │    ┌──────┤ 循环开始       │
  │         │          │    │      │              │
  │         │          │    │ 1. 读取上一个token   │
  │         │          │    │ Token Embed +      │
  │         │          │    │ Pos Encode         │
  │         │          │    │────────────────────▶│
  │         │          │    │      │              │
  │         │          │    │ 2. Transformer Forward │
  │         │          │    │    只计算新token,    │
  │         │          │    │    复用已有KV Cache  │
  │         │          │    │────────────────────▶│
  │         │          │    │      │              │
  │         │          │    │ 3. 更新 KV Cache     │
  │         │          │    │ (追加新token的KV)   │
  │         │          │    │◀────────────────────│
  │         │          │    │      │              │
  │         │          │    │ 4. 计算输出概率 &    │
  │         │          │    │    采样新token       │
  │         │          │    │────────────────────▶│
  │         │          │    │      │              │
  │         │          │    │ 5. 判断是否结束       │
  │         │          │    │◀────────────────────│
  │         │          │    │      │              │
  │         │          │    │ 是 ──┘ 否 → 继续    │
  │         │          │    │  循环结束  回到1    │
  │         │          │    │                     │
  │         │          │    └─────────────────────│
  │         │          │           │              │
  │         │          │  返回完整响应 │              │
  │         │          │◀──────────│              │
  │         │ 流式返回  │           │              │
  │◀────────│─────────│           │              │
  │         │          │           │              │
  ▼         ▼          ▼           ▼              ▼

  特点: 内存密集型 (memory-bound)
  每步只计算1个token，但需要读取完整KV Cache
  GPU计算利用率低，瓶颈在显存带宽
```
