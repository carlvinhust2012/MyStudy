# RDMA 连接与 Queue Pair分析

## 1. RDMA 是长连接还是短连接？

**RDMA 是长连接**，具体取决于使用的传输类型：

| 传输类型 | 连接模式 | 说明 |
|---------|---------|------|
| **RC (Reliable Connection)** | **长连接** | 类似 TCP，1对1可靠连接，NCCL/RoCEv2 最常用 |
| UC (Unreliable Connection) | 长连接 | 1对1不可靠连接 |
| UD (Unreliable Datagram) | 无连接 | 类似 UDP，无连接状态 |
| XRC (Extended RC) | 共享长连接 | 多个 QP 共享同一个连接上下文 |

### 连接生命周期

1. **连接建立阶段**：双方通过 TCP/SIDR 协商，创建 QP（Queue Pair），完成状态交换
2. **数据传输阶段**：连接保持，直接通过 RDMA Write/Read/Send 操作内存，无需重复建连
3. **连接维持**：直到通信结束才拆除

### 与 TCP 短连接对比

- TCP 短连接：每次请求都要三次握手 → 四次挥手
- RDMA RC：建立一次 QP 后，后续所有通信直接走硬件通道，**零拷贝、内核旁路**

NCCL 初始化时会有较明显的建连时间，但后续 allreduce/alltoall 等集合通信非常快，原因正是 RDMA 长连接的特性。

## 2. Queue Pair 是软件还是硬件？

**QP 是软硬件协同的，但核心状态和执行在硬件上。**

### 软件部分

用户态/内核态通过软件创建和管理 QP：

- `ibv_create_qp()` — 用户态调用，向驱动申请创建 QP
- 软件维护 QP 的**元数据**（QP number、状态机、配置参数等）
- 软件通过 Work Request（WR）向 QP 提交任务（Post Send/Post Recv）

### 硬件部分

真正执行数据传输的是**网卡硬件（RNIC）**：

- 网卡内部有专门的**硬件队列**（SQ 发送队列、RQ 接收队列、CQ 完成队列）
- QP 的**状态机**（RESET → INIT → RTR → RTS）由网卡硬件维护
- RDMA 操作（Read/Write/Send）由**网卡 DMA 引擎直接完成**，不经过 CPU
- 网卡内部用 **SRAM** 存储队列条目，硬件直接调度执行

```
软件: 创建/配置/提交任务    →  ibv_create_qp(), ibv_post_send()
硬件: 存储/调度/执行传输    →  RNIC 内部的 SQ/RQ/CQ + DMA 引擎
```

### 与 TCP Socket 对比

| | TCP Socket | RDMA QP |
|--|-----------|---------|
| 队列存储 | 内核内存 | **网卡 SRAM** |
| 协议处理 | CPU + 内核协议栈 | **网卡硬件** |
| 数据搬运 | CPU 拷贝（至少 2 次） | **网卡 DMA 直接搬运** |

QP 硬件化是 RDMA 能做到内核旁路和零拷贝的根本原因——数据路径上完全绕过 CPU 和内核协议栈。

## 3. 两个节点之间需要建立几个连接？

这取决于具体场景，从**最少 1 个到几十个不等**。

### 最基本的情况：1 个 RC QP

两个节点之间逻辑上只需要 1 个 QP 就可以完成双向通信：

```
Node A                          Node B
┌──────────┐    RC QP (双向)    ┌──────────┐
│  NIC     │◄──────────────────►│  NIC     │
│ SQ ──►   │     Send/Write     │   ──► SQ │
│ RQ ◄──   │◄──── Recv/Read ────│   ◄── RQ │
└──────────┘                    └──────────┘
```

一个 RC QP 本身就支持双向：SQ 发送、RQ 接收。

### 实际生产中远不止 1 个

#### 多网卡/多端口

一台 GPU 服务器通常有多张网卡（如 4x 200Gbps NIC），每张网卡至少建 1 个 QP：

```
Node A                         Node B
NIC 0 ── QP0 ─────────────── QP0 ── NIC 0
NIC 1 ── QP1 ─────────────── QP1 ── NIC 1
NIC 2 ── QP2 ─────────────── QP2 ── NIC 2
NIC 3 ── QP3 ─────────────── QP3 ── NIC 3
```

#### NCCL 场景：GPU 对 GPU 建连

NCCL 是按**GPU 之间**建连的，而不是按节点。一台 8 卡机器和另一台 8 卡机器之间：

```
Node A (8 GPUs)          Node B (8 GPUs)
GPU 0 ── QP ─────────────── GPU 0
GPU 1 ── QP ─────────────── GPU 1
...                      ...
GPU 7 ── QP ─────────────── GPU 7
```

**至少 8 个 QP**（每对 GPU 之间 1 个）。

#### 可能进一步增加的因素

- **多 QP per GPU pair**：为充分利用带宽，每对 GPU 之间可能建多个 QP
- **Control QP + Data QP 分离**：控制面和数据面分开，再加倍
- **每个 NIC 端口独立 QP**：跨多个网卡做聚合

### 典型 8 卡集群 QP 数量

| 场景 | 两节点间 QP 数量 |
|------|----------------|
| 最简：单网卡、单 QP | **1** |
| 实际 NCCL：8 卡对 8 卡，每对 1 QP | **8** |
| 多网卡 + 多 QP：4 NIC、每 GPU 对 2 QP | **64** |

---

## 4. 总结

| 问题 | 结论 |
|------|------|
| RDMA 是长连接还是短连接？ | **长连接**，NCCL 使用 RC 模式，建立一次 QP 后持续复用 |
| QP 是软件还是硬件？ | **软硬件协同**，软件创建配置，网卡硬件存储队列并执行传输 |
| 两节点间需要几个连接？ | **逻辑最小 1 个**，NCCL 中 8 卡集群至少 8 个，生产环境可达几十个 |

三者之间的关系：RDMA 长连接保证了 QP 的复用性，QP 的硬件化保证了高性能，多 QP 并发保证了高带宽——这三点共同构成了 NCCL 跨节点 RDMA 通信的性能基础。
